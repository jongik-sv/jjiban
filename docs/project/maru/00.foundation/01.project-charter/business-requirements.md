# MARU 시스템 비즈니스 제품 요구 문서
## Business Requirements Document (BRD)

**프로젝트**: MARU (Master Code & Rule Management System)
**버전**: 1.0
**작성일**: 2025-01-17
**문서 유형**: PoC (Proof of Concept)

---

## 1. 프로젝트 개요

### 1.1 프로젝트 목적
MARU(Master Code & Rule Management System)는 중앙화된 마스터 코드와 비즈니스 룰 관리를 통해 시스템 간 데이터 일관성을 보장하는 시스템입니다.

### 1.2 프로젝트 배경
- 다양한 시스템에서 사용되는 참조 코드와 비즈니스 룰이 분산 관리되어 일관성 문제 발생
- 코드/룰 변경 시 여러 시스템 수정으로 인한 운영 복잡성 증가
- 변경 이력 추적 및 버전 관리의 어려움

### 1.3 PoC 목표
- 중앙화된 마스터 데이터 관리 시스템의 기술적 타당성 검증
- 선분 이력 모델을 통한 버전 관리 시스템 구현 검증
- Frontend(Nexacro) - Backend(Node.js) - Database(Oracle) 연동 확인

---

## 2. 비즈니스 목표 및 가치

### 2.1 비즈니스 목표
- **데이터 일관성 보장**: 모든 시스템에서 동일한 참조 데이터 사용
- **운영 효율성 향상**: 중앙에서 한 번의 변경으로 전체 시스템 반영
- **변경 이력 추적**: 모든 변경사항에 대한 완전한 이력 관리
- **데이터 품질 향상**: 중앙화된 검증 및 관리 프로세스

### 2.2 기대 효과
- 시스템 간 데이터 불일치 문제 해결
- 코드/룰 변경 작업 시간 50% 단축
- 데이터 품질 관리 체계 확립
- 규정 준수 및 감사 대응 능력 향상

## 4. 기능 요구사항

### 4.1 마스터 코드 관리

#### UC-001: 코드 헤더 관리
**사용자 스토리**: 관리자로서 코드 집합의 기본 정보를 관리하고 싶다.

**기능 요구사항**:
- 마루ID 생성 및 관리
- 마루명, 타입(CODE/RULE), 상태 관리
- 데이터 우선순위 사용 여부 설정
- 버전 관리 (시작/종료 일시)

**상태 관리**:
- CREATED: 자유로운 수정 가능
- INUSE: 이력 생성을 통한 수정
- DEPRECATED: 수정 불가

#### UC-002: 코드 카테고리 관리
**사용자 스토리**: 관리자로서 코드를 카테고리별로 분류하고 검증 규칙을 정의하고 싶다.

**기능 요구사항**:
- 카테고리별 코드 분류
- 코드 타입 정의 (Normal: 콤마구분, RegEx: 정규식)
- 코드 정의 규칙 설정
- 버전 및 이력 관리

#### UC-003: 코드 기본값 관리
**사용자 스토리**: 관리자로서 실제 사용할 코드값과 설명을 관리하고 싶다.

**기능 요구사항**:
- 코드값 및 기본 설명 관리
- 다국어 지원 (대체 코드명 1~5)
- 순번 관리
- 코드 카테고리 정의와의 일관성 검증

### 4.2 비즈니스 룰 관리

#### UC-004: 룰 변수 정의
**사용자 스토리**: 관리자로서 비즈니스 룰의 조건과 결과 변수를 정의하고 싶다.

**기능 요구사항**:
- 변수 타입 관리 (OP: 조건, RSLT: 결과)
- 변수 위치 및 순서 정의
- 데이터 타입 관리 (String, Number, Boolean)
- 조건 타입 관리 (Equal, One, Two)

#### UC-005: 룰 레코드 관리
**사용자 스토리**: 관리자로서 실제 비즈니스 룰을 조건-결과 형태로 관리하고 싶다.

**기능 요구사항**:
- 조건(OP) 및 결과(RSLT) 매트릭스 관리 (각 20개까지)
- 우선순위 관리
- 룰 설명 관리
- 연산자 지원 (=, !=, <, >, <=, >=, BETWEEN, IN, NOT_IN, NOT_CHECK, SCRIPT)

### 4.3 공통 기능

#### UC-006: 이력 관리
**사용자 스토리**: 관리자로서 모든 변경사항에 대한 완전한 이력을 조회하고 싶다.

**기능 요구사항**:
- 선분 이력 모델 구현 (시작일시/종료일시)
- 버전별 변경사항 추적
- 특정 시점 데이터 조회 기능

#### UC-007: 데이터 검증
**사용자 스토리**: 시스템으로서 데이터 일관성을 자동으로 검증하고 싶다.

**기능 요구사항**:
- 코드 카테고리 정의와 실제 코드값 일치성 검증
- 중복 코드 검증
- 필수 값 검증
- 날짜 범위 검증

## 5. 비즈니스 프로세스

### 5.1 코드 생성 프로세스
1. 코드 헤더 생성 (CREATED 상태)
2. 코드 카테고리 정의
3. 코드 기본값 등록
4. 검증 및 테스트
5. 상태 변경 (INUSE)

### 5.2 코드 수정 프로세스
1. 수정 요청 접수
2. 현재 상태 확인
   - CREATED: 직접 수정
   - INUSE: 이력 생성 후 수정
   - DEPRECATED: 수정 불가
3. 변경사항 적용
4. 검증 및 배포

### 5.3 룰 관리 프로세스
1. 룰 변수 정의
2. 룰 레코드 생성
3. 우선순위 설정
4. 테스트 및 검증
5. 배포

## 6. 데이터 요구사항

### 6.1 마스터 데이터
- **코드 헤더**: 약 100-500개 예상
- **코드 카테고리**: 헤더당 평균 5-10개
- **코드 기본값**: 카테고리당 평균 10-100개
- **룰 변수**: 헤더당 평균 5-20개
- **룰 레코드**: 변수 조합에 따라 10-1000개

### 6.2 이력 데이터
- 모든 변경사항에 대한 완전한 이력 보관
- 최소 3년간 이력 데이터 유지
- 특정 시점 데이터 복원 가능

### 6.3 성능 요구사항 (PoC)
- 응답시간: 일반 조회 < 1초, 복잡 조회 < 3초
- 동시 사용자: 5명 (PoC)
- 데이터 크기: 총 10,000건 이하 (PoC)

### 7.1 품질 요구사항
