# MARU 데이터베이스 기본설계서

**프로젝트**: MARU (Master Code & Rule Management System)
**문서 버전**: 1.0
**작성일**: 2025-01-17
**문서 유형**: 기본설계서

---

## 1. 개요

### 1.1 목적
MARU 시스템의 데이터베이스 구조, 테이블 설계, 인덱스 전략 및 데이터 무결성 규칙을 정의한다.

### 1.2 범위
- 논리적 데이터 모델
- 물리적 테이블 설계
- 인덱스 및 제약조건
- 선분 이력 모델 구현

---

## 2. 데이터 모델 개요

### 2.1 설계 원칙
- **선분 이력 모델**: 모든 변경사항의 시간 추적
- **정규화**: 3NF 수준의 정규화 적용
- **성능 최적화**: 적절한 비정규화 고려
- **데이터 무결성**: 제약조건을 통한 일관성 보장

### 2.2 핵심 엔티티
1. **마루 헤더 (CODE_HEAD)**: 코드/룰 집합의 메타데이터
2. **코드 카테고리 (CODE_CATE)**: 코드 분류 및 검증 규칙
3. **코드 기본값 (CODE_BASE)**: 실제 코드값과 설명
4. **룰 변수 (RULE_VAR)**: 비즈니스 룰의 변수 정의
5. **룰 레코드 (RULE_RECORD)**: 실제 비즈니스 룰 데이터

---

## 3. 테이블 상세 설계

### 3.1 TB_MR_HEAD (마루 헤더)

#### 3.1.1 테이블 목적
코드 및 룰 집합의 기본 정보와 메타데이터를 관리

#### 3.1.2 테이블 구조
```sql
CREATE TABLE TB_MR_HEAD (
    MARU_ID         VARCHAR2(50)    NOT NULL,
    VERSION         NUMBER(10,0)    NOT NULL,
    MARU_NAME       VARCHAR2(200)   NOT NULL,
    MARU_STATUS     VARCHAR2(20)    DEFAULT 'CREATED' NOT NULL,
    MARU_TYPE       VARCHAR2(10)    NOT NULL,
    PRIORITY_USE_YN CHAR(1)         DEFAULT 'N' NOT NULL,
    START_DATE      TIMESTAMP       DEFAULT CURRENT_TIMESTAMP NOT NULL,
    END_DATE        TIMESTAMP       DEFAULT TO_TIMESTAMP('9999-12-31 23:59:59', 'YYYY-MM-DD HH24:MI:SS') NOT NULL,

    CONSTRAINT PK_TB_MR_CODE_HEAD PRIMARY KEY (MARU_ID, VERSION),
    CONSTRAINT CK_MARU_STATUS CHECK (MARU_STATUS IN ('CREATED', 'INUSE', 'DEPRECATED')),
    CONSTRAINT CK_MARU_TYPE CHECK (MARU_TYPE IN ('CODE', 'RULE')),
    CONSTRAINT CK_PRIORITY_USE_YN CHECK (PRIORITY_USE_YN IN ('Y', 'N'))
);
```

#### 3.1.3 컬럼 설명
| 컬럼명 | 데이터타입 | 설명 | 비고 |
|--------|------------|------|------|
| MARU_ID | VARCHAR2(50) | 마루 고유 식별자 | PK |
| VERSION | NUMBER(10,0) | 버전 번호 | PK, 0부터 시작 |
| MARU_NAME | VARCHAR2(200) | 마루명 | |
| MARU_STATUS | VARCHAR2(20) | 상태 | CREATED/INUSE/DEPRECATED |
| MARU_TYPE | VARCHAR2(10) | 타입 | CODE/RULE |
| PRIORITY_USE_YN | CHAR(1) | 우선순위 사용여부 | Y/N |
| START_DATE | TIMESTAMP | 시작일시 | 선분이력 |
| END_DATE | TIMESTAMP | 종료일시 | 선분이력 |

### 3.2 TB_MR_CODE_CATE (코드 카테고리)

#### 3.2.1 테이블 구조
```sql
CREATE TABLE TB_MR_CODE_CATE (
    MARU_ID         VARCHAR2(50)    NOT NULL,
    CATEGORY_ID     VARCHAR2(50)    NOT NULL,
    VERSION         NUMBER(10,0)    NOT NULL,
    CATEGORY_NAME   VARCHAR2(200)   NOT NULL,
    CODE_TYPE       VARCHAR2(20)    NOT NULL,
    CODE_DEFINITION VARCHAR2(4000)  NOT NULL,
    START_DATE      TIMESTAMP       DEFAULT CURRENT_TIMESTAMP NOT NULL,
    END_DATE        TIMESTAMP       DEFAULT TO_TIMESTAMP('9999-12-31 23:59:59', 'YYYY-MM-DD HH24:MI:SS') NOT NULL,

    CONSTRAINT PK_TB_MR_CODE_CATE PRIMARY KEY (MARU_ID, CATEGORY_ID, VERSION),
    CONSTRAINT FK_CODE_CATE_HEAD FOREIGN KEY (MARU_ID, VERSION)
        REFERENCES TB_MR_HEAD (MARU_ID, VERSION),
    CONSTRAINT CK_CODE_TYPE CHECK (CODE_TYPE IN ('Normal', 'RegEx'))
);
```

#### 3.2.2 컬럼 설명
| 컬럼명 | 데이터타입 | 설명 | 비고 |
|--------|------------|------|------|
| MARU_ID | VARCHAR2(50) | 마루 ID | PK, FK |
| CATEGORY_ID | VARCHAR2(50) | 카테고리 ID | PK |
| VERSION | NUMBER(10,0) | 버전 번호 | PK |
| CATEGORY_NAME | VARCHAR2(200) | 카테고리명 | |
| CODE_TYPE | VARCHAR2(20) | 코드 타입 | Normal/RegEx |
| CODE_DEFINITION | VARCHAR2(4000) | 코드 정의 | 콤마구분 또는 정규식 |

## 5. 선분 이력 모델 구현

### 5.1 이력 관리 원칙
- 모든 테이블에 START_DATE, END_DATE 필수
- 삭제는 논리적 삭제 (END_DATE 갱신)
- 수정 시 이전 레코드 종료 후 신규 레코드 생성
- 특정 시점 데이터 조회 가능

### 5.2 상태별 처리 방식
```sql
-- CREATED 상태: 직접 수정
UPDATE TB_MR_HEAD
SET MARU_NAME = :newName
WHERE MARU_ID = :maruId AND END_DATE = TO_TIMESTAMP('9999-12-31 23:59:59', 'YYYY-MM-DD HH24:MI:SS');

-- INUSE 상태: 이력 생성 후 수정
-- 1. 기존 레코드 종료
UPDATE TB_MR_HEAD
SET END_DATE = :currentTime
WHERE MARU_ID = :maruId AND END_DATE = TO_TIMESTAMP('9999-12-31 23:59:59', 'YYYY-MM-DD HH24:MI:SS');

-- 2. 신규 레코드 생성
INSERT INTO TB_MR_HEAD (MARU_ID, VERSION, MARU_NAME, ...)
SELECT MARU_ID, VERSION + 1, :newName, ...
FROM TB_MR_HEAD
WHERE MARU_ID = :maruId AND END_DATE = :currentTime;
```

## 6. 데이터 무결성 규칙

### 6.1 참조 무결성
- 모든 자식 테이블은 부모 테이블 참조
- CASCADE DELETE 제한 (논리적 삭제 사용)

### 6.2 도메인 무결성
- CHECK 제약조건으로 허용값 제한
- NOT NULL 제약조건으로 필수값 보장

### 6.3 비즈니스 규칙
- 코드 카테고리 정의와 코드 기본값 일치성 검증
- 룰 변수와 룰 레코드 간 일관성 검증
