### 🔴 P1 - 즉시 해결 필요
**[중복 요청 방지 미비로 인한 중복 생성 위험]**
- **심각도**: High
- **카테고리**: 품질/아키텍처
- **설명**: 신규 생성/저장/상태 전이 등 CUD 액션에 대해 더블클릭/빠른 연속 클릭 방지, 요청 중 버튼 비활성, in-flight 요청 취소 정책이 명문화되어 있지 않음.
- **영향**: 동일 엔티티 중복 생성·경합으로 서버 에러/데이터 불일치 발생, 사용자 혼란.
- **개선안**: 전송 단위로 글로벌 submitting 플래그/버튼 디스에이블 + 스피너 적용, 동일 키(idempotency key) 정책(요청 GUID 헤더) 문서화. Axios 취소 토큰 유사 메커니즘으로 이전 요청 취소.

### 🟠 P2 - 빠른 해결 권장
**[입력 검증/정규화 명세 세분화 필요]**
- **심각도**: Medium
- **카테고리**: 품질/보안
- **설명**: ds_maruDetail 필드별 길이/패턴/허용값/필수 여부 표가 요약적. 백엔드(Joi)와의 스키마를 UI 문서에 표준 표로 동기화 필요.
- **개선안**: MARU_ID(패턴/최대길이), MARU_NAME(길이/공백 규칙), MARU_TYPE(열거형 CODE/RULE), STATUS(열거형), DATE 필드(형식/타임존) 등 테이블로 명시.

**[에러 모델/상태코드 매핑의 UI 반영 기준 부족]**
- **심각도**: Medium
- **카테고리**: 품질
- **설명**: 서버 표준 에러 포맷을 UI 메시지로 변환하는 규칙(필드 단위 vs 글로벌, 재시도 가능 여부, 포커스 이동)이 정형화되어 있지 않음.
- **개선안**: 오류코드→메시지 맵, 포커스 정책, 재시도/취소 가이드, 토스트/다이얼로그 사용 기준 정의. 실패 예시를 Playwright 시나리오와 연동.

**[REST 콘텐츠 협상(JSON/Nexacro XML) 대응 명확화]**
- **심각도**: Medium
- **카테고리**: 아키텍처
- **설명**: 백엔드는 Nexacro Dataset XML 응답을 지원하나, UI 문서에서 Accept/Content-Type 협상 기준 및 변환 계층의 책임이 모호.
- **개선안**: 기본 JSON, 필요 시 XML 변환 어댑터 계층 명시. Dataset <-> JSON 매핑 테이블 제공.

**[검색/필터 UX 성능 개선(디바운스/즉시반영 정책)]**
- **심각도**: Medium
- **카테고리**: 성능/UX
- **설명**: 검색어 입력 시 즉시반영/버튼트리거 혼재. 데이터가 증가할 경우 불필요 요청 다발 가능.
- **개선안**: 입력 300ms 디바운스 + 엔터키/버튼 확정 조합 정책, 최소 글자 수 조건, 진행 중 인디케이터.

### [ID-001] CUD 액션 중복 요청 방지 (P1)
- **원본**: 더블클릭/연속 클릭 방지, in-flight 요청 취소 정책 미비
- **방식**: 📝 조정 적용
- **내용**:
  - 저장/삭제/상태변경 버튼 클릭 즉시 비활성화 (`btn.enable = false`)
  - Transaction 완료 시 재활성화
  - idempotency key는 프로덕션 대비 (PoC 제외)
- **섹션**: §5.1 프로세스 흐름 (신규/수정/삭제 단계), §8.0 공통 Transaction 처리 패턴
- **이유**: PoC 단일 사용자 환경이나 UX 품질 향상, Nexacro 기본 기능으로 간단 구현 가능

### [ID-002] 검증/정규화 스키마 상세 명세 (P2)
- **원본**: 필드별 길이/패턴/허용값 표준 표로 명시 필요
- **방식**: ✅ 그대로 적용
- **내용**:
  - MARU_ID: `^[A-Za-z0-9_]{1,20}$`, 최대 20자
  - MARU_NAME: 1-100자, 특수문자 제한 (`<>` 금지)
  - MARU_TYPE: 열거형 `CODE|RULE`
  - 날짜 필드: YYYYMMDDHHmmss 형식
- **섹션**: §7.4 유효성 검증 규칙 (표 확장)
- **이유**: 개발자 구현 가이드 명확화, Backend Joi 스키마와 동기화

### [ID-004] 에러 모델/상태코드 매핑 (P2)
- **원본**: 서버 에러를 UI 메시지로 변환하는 규칙 정형화
- **방식**: ✅ 그대로 적용
- **내용**:
  - ErrorCode → 사용자 메시지 매핑 표 (12개 항목)
  - 필드 단위(Field) vs 글로벌(Global) 오류 구분
  - 포커스 이동 정책 명시
  - 재시도 정책 (자동/수동) 구분
- **섹션**: §9.1 예상 오류 상황 및 처리 방안 (매핑 표 추가)
- **이유**: 일관된 UX, 테스트 케이스와 연동

### [ID-006] 환경설정/엔드포인트 관리 (P3)
- **원본**: baseURL, timeout, 재시도 등 환경 설정 분리
- **방식**: ✅ 그대로 적용
- **내용**:
  - Nexacro 전역 변수로 설정 정의
  - `application.gv_baseURL = "http://localhost:3000"`
  - `application.gv_timeout = 30000`
  - `application.gv_retryCount = 2`
  - 문서에 설정 항목 표 추가
- **섹션**: §4.2 외부 의존성 (환경 설정 관리 표)
- **이유**: 환경별 배포 용이, 모범 사례

### [ID-009] 접근성 체크리스트 정교화 (P4)
- **원본**: Tab 순서, 포커스 링, aria-label 강화
- **방식**: ✅ 그대로 적용
- **내용**:
  - 포커스 순서 5단계 명시 (검색 → 액션 → 그리드 → 상세 → 하단버튼)
  - 주요 필드 aria-label 추가 (edt_maruId, edt_maruName, btn_checkDup)
  - aria-invalid 적용 (유효성 검증 실패 시)
  - 오류 메시지 영역에 role="alert"
  - 키보드 단축키 일관성 계획 (Ctrl+N, Ctrl+S는 향후 확장)
- **섹션**: §6.3 접근성 가이드 (포커스 순서, ARIA 속성)
- **이유**: 웹 표준 준수, Nexacro 기본 지원 활용
