## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[보안] 클라이언트 측 입력 검증 의존 위험**
- **심각도**: Critical
- **카테고리**: 🛡️ 보안
- **설명**: 상세설계서 10절 '보안/품질 고려' 항목에 "XSS 방지: 특수문자 필터링"이 명시되어 있으나, 이것이 클라이언트 측(Nexacro)에서만 처리되는지 또는 서버 측에서도 강제되는지 불분명합니다. 만약 클라이언트 측 검증에만 의존한다면, 악의적인 사용자가 API를 직접 호출하여 검증을 우회하고 XSS 또는 기타 페이로드 공격을 시도할 수 있습니다.
- **영향**: 악성 스크립트 실행으로 인한 세션 탈취, 데이터 변조 등의 심각한 보안 사고로 이어질 수 있습니다.
- **개선안**: 상세설계서에 **"모든 입력값에 대한 최종 검증(Validation) 및 정제(Sanitization)는 반드시 서버 측(Backend)에서 수행되어야 함"**을 명시해야 합니다. 클라이언트 측 검증은 사용자 편의성을 위한 보조 수단으로만 사용되어야 함을 명확히 해야 합니다.
- **참조**: OWASP Top 10 - A03:2021-Injection

### 🟠 P2 - 빠른 해결 권장
**[성능] 데이터 변경 후 목록 전체 재조회로 인한 비효율**
- **심각도**: High
- **카테고리**: ⚡ 성능
- **설명**: 상세설계서 5.2 프로세스 설계 개념도에서 변수 저장(RV003/RV004) 또는 삭제(RV005) 후, '목록 재조회(O)' 단계로 이어집니다. 이는 매번 데이터 변경 시마다 전체 목록을 다시 조회하는 방식으로, 변수 목록이 많아질 경우 불필요한 네트워크 트래픽과 서버 부하를 유발하며 UI 응답성을 저하시킵니다.
- **영향**: 데이터가 증가함에 따라 시스템 성능이 저하되고 사용자 경험이 나빠집니다.
- **개선안**:
  1.  **생성/수정 시**: API 응답 본문에 포함된 단일 객체 데이터(생성/수정된 변수 정보)를 사용하여 클라이언트 측 Dataset(ds_ruleVarList)의 해당 행만 직접 추가하거나 수정합니다.
  2.  **삭제 시**: API 호출 성공 시, 클라이언트 측 Dataset에서 해당 행을 즉시 제거합니다.
  3.  이를 통해 불필요한 전체 목록 재조회 API 호출을 제거하여 성능을 최적화할 수 있습니다.
- **참조**: 상세설계서 5.2 Mermaid 다이어그램의 `L --> O`, `N --> O` 경로

### 🟡 P3 - 보통 우선순위
**[아키텍처] Nexacro 종속적인 API 응답 구조**
- **심각도**: Medium
- **카테고리**: 🏗️ 아키텍처
- **설명**: `api-design.md`에 따르면 모든 API 응답이 Nexacro의 Dataset XML 형식에 맞춰져 있고, 실제 성공/실패 여부와 관계없이 HTTP 상태 코드는 항상 `200 OK`로 반환됩니다. 이는 시스템의 유연성과 재사용성을 저해하는 설계입니다.
- **영향**: 향후 Nexacro가 아닌 다른 클라이언트(예: 웹 대시보드, 모바일 앱, 외부 연동 시스템)에서 해당 API를 사용하고자 할 때, 불필요한 XML 파싱과 비표준적인 에러 처리 로직을 구현해야 하는 부담이 발생합니다.
- **개선안**:
  1.  성공, 실패, 권한 없음 등을 나타내는 표준 HTTP 상태 코드(200, 201, 400, 401, 404, 500 등)를 사용하도록 API 설계를 변경하는 것을 장기적으로 권장합니다.
  2.  응답 형식은 범용적인 JSON을 기본으로 하고, 요청 헤더의 `Accept` 값에 따라 Nexacro용 XML을 선택적으로 반환하는 콘텐츠 협상(Content Negotiation) 도입을 고려해볼 수 있습니다.
- **참조**: `docs/project/maru/00.foundation/02.design-baseline/3. api-design.md` 섹션 3, 10

### 🟢 P4 - 개선 항목
**[아키텍처] 모호한 반응형 UI 전략**
- **심각도**: Low
- **카테고리**: 🏗️ 아키텍처
- **설명**: 상세설계서 6.3절에 Desktop, Tablet, Mobile에 대한 반응형 레이아웃이 언급되어 있습니다. 하지만 프로젝트의 전반적인 아키텍처와 UI 디자인 기준은 Nexacro N 기반의 데스크톱 환경에 집중되어 있습니다. 태블릿/모바일 지원에 대한 구체적인 계획이나 가이드라인 없이 개별 화면에서만 반응형을 고려하는 것은 전체적인 사용자 경험의 일관성을 해칠 수 있습니다.
- **영향**: 향후 모바일/태블릿 지원 시 중복 개발이나 UI/UX 파편화 문제가 발생할 수 있습니다.
- **개선안**: 프로젝트 차원에서 모바일/태블릿 지원 범위를 명확히 하고, 이에 대한 UI/UX 가이드라인을 수립하여 모든 화면 설계에 일관되게 적용할 것을 제안합니다. 현 단계에서는 데스크톱 전용임을 명확히 하고, 반응형 설계는 미래 확장 고려사항으로만 기술하는 것이 혼선을 줄일 수 있습니다.
- **참조**: 상세설계서 6.3절 '반응형/접근성/상호작용 가이드'
