## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[SCRIPT 실행에 대한 보안·샌드박스 전략 부재]**
- **심각도**: Critical · **카테고리**: 보안/아키텍처
- **설명**: 조건/결과 종류에 SCRIPT 타입을 허용하지만, 실행 엔진, 샌드박스, 타임아웃, 화이트리스트 등 보안 장치가 설계에 없다.
- **영향**: 악성 스크립트 실행, 시스템 제어권 탈취, 데이터 유출 가능.
- **개선안**: (1) SCRIPT 지원 범위 축소 또는 금지, (2) 샌드박스(별도 프로세스/VM) 및 자원 제한, (3) 승인 워크플로우/코드 리뷰, (4) 로깅·추적·알림 체계와 Joi/Swagger 제한 반영.

### 🟠 P2 - 빠른 해결 권장
**[PRIORITY_ORDER 재배치 동시성/락 정책 불명확]**
- 여러 사용자가 동시에 우선순위 변경 시 순번이 엇갈릴 수 있음. → 트랜잭션 범위, 낙관적 잠금, 일괄 업데이트 전략 명시 필요.

**[OP/RSLT 필드 Validation 세분화]**
- 20개의 컬럼에 대한 허용 값·길이·필수 여부·데이터 타입(숫자/문자/Boolean) 규칙이 표준화되지 않음. → Joi/Swagger에 Case별 스키마 추가 필요.

**[에러 모델/로깅 표준화 부족]**
- ErrorCode/Message 구조가 부분적으로만 기술. → HTTP 상태, 재시도 가능 여부, 사용자 메시지/개발자 메시지 분리, 감사 로그 항목 정의 필요.

### 🟡 P3 - 보통 우선순위
- **캐시/성능**: 다중 조건 조합 조회에 대한 인덱스 전략((MARU_ID, VERSION, PRIORITY_ORDER), (MARU_ID, CONDITION_HASH)), 캐시 키/TTL 정의 필요.
- **데이터 품질**: OP/RSLT 컬럼에 NULL 허용 여부, 스파스 데이터 처리 정책(빈 컬럼 정리)이 불명확.
- **테스트**: 20개 필드 경계값, PRIORITY 중복, SCRIPT 악성 입력 등 자동화 시나리오 보강 필요.

### 🟢 P4 - 개선 항목
- 배포 전 검증용 Rule Simulator API/스크립트 제공 검토.
- 규칙 변경 이력/승인 플로우/롤백 전략 문서화.

### 🔵 P5 - 참고 사항
- RL0300(실행)과 연계해 캐시/통계/알림 설계 미리 협의하면 향후 통합이 수월.
