## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[스키마-설계 불일치: DESCRIPTION 필드]**
- **심각도**: High
- **카테고리**: 아키텍처/품질
- **설명**: 상세설계와 일부 문서/구현 흐름에서 `DESCRIPTION` 필드를 전제로 하나, 실제 테이블(TB_MR_HEAD)에는 해당 컬럼이 없음. 구현부에서도 주석 처리 흔적 존재.
- **영향**: UI/서비스 계층에서 설명 필드 사용 시 데이터 손실/에러 가능. API/스키마/문서 간 불일치로 테스트 실패 및 유지보수 비용 증가.
- **개선안**:
  1) 스키마에 `DESCRIPTION VARCHAR2(400)`(또는 요구 길이) 추가하는 마이그레이션 제공, OR
  2) 상세설계/Swagger/DTO에서 `description` 항목을 제거하고 전 구간 일관성 재정렬.
  선택 시 기준: 실제 사용성(요구사항/UX)과 기존 3.2에서 보고된 이슈를 반영해 결정. 어느 경우든 문서/코드/테스트 동시 수정.
- **참조**: backend/src/repositories/MaruHeaderRepository.js:361,381 (설명 컬럼 주석 처리)

### 🟠 P2 - 빠른 해결 권장
**[보안 기초 미적용(인증/인가, HTTPS)]**
- **심각도**: High
- **카테고리**: 보안
- **설명**: PoC 제약으로 인증/인가/HTTPS 미적용. 설계서에 정책/확장 전략 부재.
- **영향**: 운영 전환 시 주요 취약점으로 직결. 감사/규정 대응 불가.
- **개선안**: 최소 JWT/OAuth2 기반 인증·인가 도입 계획과 역할별 권한 모델 정의, HTTPS 종단 및 시크릿 관리(.env) 운영 가이드 추가. Swagger 보안 스키마 명시.

**[상태 전이/버전 처리의 트랜잭션 경계 모호]**
- **심각도**: Medium
- **카테고리**: 아키텍처/품질
- **설명**: INUSE 상태에서의 변경 시 신규 버전 생성 규칙, 동시 업데이트 경쟁 조건(동일 MARU_ID 동시 갱신) 처리 방안이 서술적으로만 제시됨.
- **영향**: 경합 상황에서 데이터 무결성 훼손 가능. 성능 저하/교착 위험.
- **개선안**: (MARU_ID, VERSION) 유니크키 + 낙관적 락(버전/업데이트 타임스탬프) 또는 DB 락 전략 명시. 전이 API(MH006)와 버전 생성 로직의 단일 트랜잭션 보장.

**[REST 일관성/콘텐츠 협상]**
- **심각도**: Medium
- **카테고리**: 아키텍처/품질
- **설명**: 요청 JSON, 응답 Nexacro XML 병행. 시스템 맥락상 필요할 수 있으나 Accept/Content-Type 협상 정책 미정의.
- **영향**: 클라이언트 다양성 증가 시 유지보수성 하락. 테스트 복잡도 증가.
- **개선안**: 기본 응답 JSON, `Accept: application/xml` 시 Nexacro Dataset XML 제공 등 명시. Swagger에 두 응답 스키마 등록.

**[인덱스/쿼리 성능 계획 미흡]**
- **심각도**: Medium
- **카테고리**: 성능
- **설명**: 목록/조건 조회 목표 제시(1s/500ms) 대비 인덱스 설계 명시가 부족.
- **영향**: 데이터 증가 시 성능 목표 미달 위험.
- **개선안**: TB_MR_HEAD에 (MARU_ID), (STATUS), (START_DATE, END_DATE), (CREATED_AT) 등 사용 패턴 기반 인덱스 명시. 실행계획과 함께 성능 검증 항목 추가.

### 🟡 P3 - 보통 우선순위
**[에러 모델/상태코드 표준화 부족]**
- **심각도**: Medium
- **카테고리**: 품질
- **설명**: 오류 응답 포맷(코드/메시지/원인/추적ID) 및 HTTP 상태코드 매핑 표 미흡.
- **개선안**: 에러 카탈로그 및 표준 응답 바디(JSON) 정의. Swagger에 예시 포함.

**[검증(Validation) 커버리지 명확화]**
- **심각도**: Low
- **카테고리**: 품질/보안
- **설명**: Joi 사용 언급은 있으나 필드별 스키마(길이/패턴/허용값) 표가 부족.
- **개선안**: DTO 단위 Joi 스키마 표(REQ-001~006 대응) 추가, 경계값 테스트 항목 연동.

**[Swagger 문서 완성도]**
- **심각도**: Low
- **카테고리**: 품질
- **설명**: 문서화 계획은 있으나 모든 API(MH001~MH008)의 파라미터/응답 스키마/예외 케이스 예시가 누락 가능.
- **개선안**: 성공/실패 예시, 페이징/정렬 파라미터, 상태 전이 제약 조건을 문서에 반영.

### 🟢 P4 - 개선 항목
**[캐싱 전략 초안]**
