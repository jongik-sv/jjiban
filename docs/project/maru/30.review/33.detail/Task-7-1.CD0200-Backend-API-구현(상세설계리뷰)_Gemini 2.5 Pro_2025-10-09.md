## ⚠️ 발견된 이슈

### 🟠 P2 - 빠른 해결 권장

**[이슈 제목] 데이터베이스 스키마 불일치 및 비효율적 데이터 타입 사용**
- **심각도**: High
- **카테고리**: 🏗️ 아키텍처
- **설명**: 상세설계서(§7.3)에 정의된 `TB_MR_CODE_BASE` 테이블의 컬럼 타입이 데이터베이스 기본설계서(§3.3)와 일치하지 않습니다. 특히, `START_DATE`, `END_DATE` 등의 날짜/시간 관련 필드가 `VARCHAR2(14)`로 설계되어 있습니다. 이는 데이터 정합성, 성능, 유지보수 측면에서 심각한 문제를 야기할 수 있습니다.
- **영향**:
  - 날짜 비교 및 범위 검색 시 `TO_DATE` 함수를 사용해야 하므로 인덱스를 활용하지 못해 심각한 성능 저하 발생.
  - 유효하지 않은 날짜 형식(예: '20251340')이 입력될 수 있어 데이터 무결성이 훼손됨.
  - 날짜 계산 및 시간대(Timezone) 처리가 복잡하고 오류 발생 가능성이 높음.
- **개선안**: 데이터베이스 기본설계서와 같이 `TB_MR_CODE_BASE` 테이블의 `START_DATE`, `END_DATE`, `CREATE_DATE`, `UPDATE_DATE` 컬럼 타입을 `TIMESTAMP` 또는 `DATE`로 변경해야 합니다. API 응답 시점에 `YYYYMMDDHH24MISS` 형식의 문자열로 변환하여 전달하도록 응용 프로그램 로직을 수정합니다.
- **참조**: `docs/project/maru/00.foundation/02.design-baseline/2. database-design.md` §3.1.2

### 🟡 P3 - 보통 우선순위

**[이슈 제목] API 응답 데이터에 불필요한 정보 포함**
- **심각도**: Medium
- **카테고리**: ⚡ 성능
- **설명**: 코드 기본값 목록 조회 API(CB001)의 응답에 `START_DATE`, `END_DATE`, `VERSION` 등 UI 표시에 직접적으로 필요하지 않은 내부 관리용 데이터가 포함되어 있습니다. (상세설계서 §7.2)
- **영향**:
  - 불필요한 데이터 전송으로 네트워크 대역폭 낭비 및 응답 시간 증가.
  - 클라이언트(Nexacro) 측에서 불필요한 데이터를 처리해야 하는 부담 발생.
  - 시스템 내부 구조(버전, 이력 관리 방식)가 외부에 노출될 수 있음.
- **개선안**: API 응답용 DTO(Data Transfer Object)를 별도로 정의하여, UI 표시에 꼭 필요한 `CODE`, `CODE_NAME`, `SORT_ORDER`, `USE_YN`, `ALTER_CODE_NAME` 등의 필드만 포함하여 반환하도록 수정합니다. 내부 관리 데이터는 응답에서 제외합니다.
- **참조**: 상세설계서 §8.1

**[이슈 제목] 일관성 없는 인증/인가 처리 방안**
- **심각도**: Medium
- **카테고리**: 🛡️ 보안
- **설명**: 상세설계서 §10.1에서 "향후 JWT 기반 인증 적용 예정"이라고 명시했으나, 기술 요구 문서(`technical-requirements.md` §7.1)에서는 "기본 세션 기반"으로 PoC 인증 수준을 정의하고 있습니다. 두 문서 간 인증 방식에 대한 계획이 일치하지 않습니다.
- **영향**:
  - 개발팀 내에서 인증 방식에 대한 혼란을 야기하여 잘못된 구현으로 이어질 수 있음.
  - PoC 이후 단계에서 인증 시스템 전환 시 추가적인 리소스가 소요될 수 있음.
- **개선안**: PoC 단계의 인증 방식을 '세션 기반' 또는 'JWT 기반' 중 하나로 명확히 통일하고, 관련 문서(상세설계서, 기술 요구 문서)의 내용을 일치시켜야 합니다. 장기적 관점에서 확장성을 고려한다면 JWT를 PoC부터 적용하는 것을 권장합니다.
- **참조**: 상세설계서 §10.1, `technical-requirements.md` §7.1

### 🟢 P4 - 개선 항목

**[이슈 제목] 하드코딩된 에러 메시지**
- **심각도**: Low
- **카테고리**: 📝 품질
- **설명**: API 오류 응답 예시(§8.1, §8.3 등)에서 "해당 마루를 찾을 수 없습니다", "코드 기본값이 정상적으로 생성되었습니다." 와 같은 메시지가 문자열 리터럴로 직접 사용되고 있습니다.
- **영향**:
  - 향후 다국어 지원 또는 메시지 중앙 관리가 어려워짐.
  - 동일한 메시지가 여러 곳에서 중복 사용될 경우 일관성 유지가 힘듦.
- **개선안**: 에러 코드 및 메시지를 별도의 상수 파일이나 설정 파일(예: `constants/messages.js`)에서 중앙 관리하도록 리팩토링합니다. API에서는 해당 코드값을 참조하여 메시지를 반환하도록 구현합니다.
- **참조**: 상세설계서 §8.1, §8.3

## 🎯 종합 평가
- **전반적인 설계 품질**: 문서의 구조와 내용이 매우 체계적이고 상세하여 구현에 필요한 정보를 충분히 제공합니다. 특히 요구사항 추적성과 테스트 전략이 우수합니다.
- **구현 준비도**: '높음'. 발견된 이슈들은 설계의 근간을 흔드는 수준이 아니며, 대부분 빠른 수정이 가능합니다. P2 이슈(데이터 타입) 해결 후 즉시 구현 착수가 가능할 것으로 판단됩니다.
- **주요 리스크**: 데이터베이스 스키마 불일치(P2)는 즉시 해결하지 않을 경우, 프로젝트 후반에 큰 기술 부채로 작용할 위험이 있습니다.
