## ⚠️ 발견된 이슈

- 해당 없음

### 🟠 P2 - 빠른 해결 권장
**[이슈-001] CSRF 방어 부재**
- **심각도**: High
- **카테고리**: 보안
- **설명**: 상세설계서 10.1절에서 CSRF 방지를 프로덕션에서 고려한다고 언급했지만, PoC 단계에서도 기본적인 방어 메커니즘(예: SameSite 쿠키 설정, Origin 헤더 검증) 적용이 필요합니다.
- **영향**: 악의적인 사이트에서 사용자의 권한을 도용하여 의도치 않은 작업을 수행할 위험이 있습니다.
- **개선안**: Backend에서 `cors` 미들웨어 설정을 강화하고, 필요시 CSRF 토큰 패턴을 적용하는 방안을 검토해야 합니다.

### 🟡 P3 - 보통 우선순위
**[이슈-002] 클라이언트 측 과도한 재조회 로직**
- **심각도**: Medium
- **카테고리**: 성능
- **설명**: 프로세스 흐름(5.1절)에서 생성/수정/삭제 등 모든 작업 후 전체 목록을 다시 조회(`GET /api/v1/maru-headers`)하고 있습니다. 이는 불필요한 부하를 유발할 수 있습니다.
- **영향**: 데이터가 많아질 경우 사용자 경험(UX) 저하 및 서버 부하 증가의 원인이 됩니다.
- **개선안**: API 응답으로 변경된 행의 데이터만 받아 `ds_maruList` Dataset을 클라이언트에서 직접 갱신(추가/수정/삭제)하는 로직으로 변경하는 것을 권장합니다.

**[이슈-003] 하드코딩된 메시지**
- **심각도**: Low
- **카테고리**: 품질
- **설명**: 오류/성공 메시지가 설계서에 하드코딩되어 있습니다. (예: "이미 존재하는 마루ID입니다.")
- **영향**: 향후 다국어 지원 또는 메시지 중앙 관리에 어려움이 있습니다.
- **개선안**: 메시지를 코드화하여 별도의 메시지 파일 또는 테이블에서 관리하는 체계를 도입하는 것을 권장합니다.

### 🟢 P4 - 개선 항목
**[이슈-004] 디바운싱(Debouncing) 미적용**
- **심각도**: Low
- **카테고리**: 성능/품질
- **설명**: 검색어 입력(`edt_searchWord`) 시 실시간 검색이 아닌 '검색' 버튼 클릭으로 동작하지만, 향후 실시간 검색으로 변경 시 디바운싱 로직이 필요합니다.
- **영향**: 실시간 검색 적용 시 불필요한 API 호출이 과도하게 발생할 수 있습니다.
- **개선안**: 검색어 입력 `onchanged` 이벤트에 300ms 정도의 디바운싱을 적용하여 API 호출을 최적화하는 방안을 미리 고려해두는 것이 좋습니다.
