## ⚠️ 발견된 이슈

### 🟠 P2 - 빠른 해결 권장
**[이슈 제목] API 인터페이스 불일치: 우선순위 재정렬 API 누락**
- **심각도**: High
- **카테고리**: 🏗️ 아키텍처
- **설명**: 상세설계서(§8.3)에는 우선순위 재정렬을 위해 `PATCH /api/v1/maru-headers/{maruId}/records/reorder` API를 호출하도록 명시되어 있습니다. 하지만 참조 문서인 `api-design.md`에는 해당 API 엔드포인트가 정의되어 있지 않습니다.
- **영향**: 프론트엔드에서 우선순위 변경 기능을 구현하더라도 연동할 백엔드 API가 존재하지 않아 기능 구현이 중단됩니다.
- **개선안**: 백엔드 API 명세(`api-design.md`)에 해당 `PATCH` API 정의를 추가하고, 백엔드 팀과 인터페이스 계약을 확정해야 합니다. 또는, 프론트엔드에서 개별 레코드의 `PUT` API를 여러 번 호출하는 방식으로 재정렬 로직을 수정해야 합니다. (전자를 권장)
- **참조**: 상세설계서 §8.3, `api-design.md`

### 🟡 P3 - 보통 우선순위
**[이슈 제목] 생성(Create)과 수정(Update) 로직의 과도한 결합**
- **심각도**: Medium
- **카테고리**: 📝 품질
- **설명**: 룰 레코드 저장 함수 `fn_saveRuleRecord`(§8.2)가 내부적으로 `RECORD_ID`의 존재 여부에 따라 생성(POST)과 수정(PUT)을 분기하여 처리하고 있습니다. 이는 단일 함수가 두 가지 책임을 갖게 되어 SRP(단일 책임 원칙)에 위배될 소지가 있습니다.
- **영향**: 코드의 가독성이 저하되고, 향후 생성과 수정 로직이 복잡해질 경우 유지보수가 어려워질 수 있습니다.
- **개선안**: `fn_createRecord`와 `fn_updateRecord` 함수로 분리하고, 두 함수가 공통으로 사용하는 유효성 검사나 데이터 구성 로직은 별도의 내부 함수로 추출하는 리팩토링을 권장합니다.
- **참조**: 상세설계서 §8.2

### 🟢 P4 - 개선 항목
**[이슈 제목] API ID 표기법 불일치**
- **심각도**: Low
- **카테고리**: 📝 품질
- **설명**: 상세설계서 전반에 걸쳐 백엔드 API를 `RR001`~`RR006`과 같은 ID로 지칭하고 있으나, 참조 문서인 `api-design.md`에는 이러한 ID 체계가 명시되어 있지 않습니다.
- **영향**: 개발자 간의 커뮤니케이션 시 혼선을 유발할 수 있으며, 문서 간의 일관성을 저해합니다.
- **개선안**: `api-design.md` 문서에 각 API의 고유 ID(예: RR001)를 명시하거나, 상세설계서에서 API ID 대신 `GET /records`와 같이 HTTP 메서드와 엔드포인트로 통일하여 지칭하는 것을 권장합니다.
- **참조**: 상세설계서 §2.1, §8
