## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[RegEx 실시간 검증 UI의 안전장치 부족]**
- **심각도**: High
- **카테고리**: 성능/UX/보안
- **설명**: RegEx 유형 입력 시 실시간 검증/미리보기 동작을 가정하지만, 복잡한 패턴 입력에 대한 시간 제한, 예외 처리, UI 비활성화 기준이 명시되지 않음.
- **영향**: Catastrophic Backtracking으로 프런트가 멈추거나 브라우저가 다운될 위험. 악의적 패턴 입력으로 서비스 거부 가능.
- **개선안**: 패턴 길이/복잡도 제한, 200ms 이상 소요 시 타임아웃/경고, 백엔드 검증 위임 옵션, 비동기 Worker 사용, 실패 시 안전한 오류 메시지 제공.

### 🟠 P2 - 빠른 해결 권장
**[트리/그리드 계층 UX 기준 부족]**
- **설명**: 부모-자식 트리를 Grid로 표현한다고만 되어 있으며, 드래그/재정렬/선택 시 경계(순환 방지, 최대 depth) 안내가 없음.
- **개선안**: 허용 depth, 순환 시 경고, 다중 선택/이동 UX 정의, Backend 무결성 규칙과 동일한 체크 추가.

**[검증/에러 메시지 표준화 필요]**
- **설명**: 필수 입력/중복/상태 전이 금지 시 메시지 예시가 부족, 영어·한글 혼용 가능성.
- **개선안**: 코드별 메시지 테이블, 포커스 이동 규칙, 재시도/취소 UX를 명시하고 공통 메시지 리소스화.

**[디바운스·요청 병합 기준 미흡]**
- **설명**: 검색·필터·상태 변경 후 즉시 재조회하지만 요청 병합/디바운스 정책이 없어 불필요 트래픽과 깜빡임 우려.
- **개선안**: 텍스트 입력 300ms 디바운스, 버튼 확정형 패턴, 전송 중 버튼 비활성, in-flight 요청 취소.

### 🟡 P3 - 보통 우선순위
**[Dataset 스키마/JSON 매핑표 필요]**
- 각 필드 타입/길이/널 허용, RegEx 패턴·설명 필드 길이 등 구체화 필요.

**[접근성 세부 기준]**
- Grid 셀 aria-label, 팝업 접근 통제, 캘린더 키보드 조작, 단축키 안내를 문서화.

### 🟢 P4 - 개선 항목
- 로딩/빈 상태/오류 전용 컴포넌트 정의 및 재사용.
- 행동 로깅(생성/상태 전이/삭제/RegEx 검증 결과)으로 운영 가시성 확보.

### 🔵 P5 - 참고 사항
- 6.1과 동일한 규칙(부모-자식 무결성, RegEx 제한)을 UI에서 선검증하여 서버 부하 감소 고려.
