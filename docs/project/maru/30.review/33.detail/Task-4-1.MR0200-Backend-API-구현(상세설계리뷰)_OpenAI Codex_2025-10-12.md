## ⚠️ 발견된 이슈

### 🟠 P2 - 빠른 해결 권장
1) [API 계약 불일치: XML vs JSON, 엔드포인트 누락]
- 심각도: High | 카테고리: 아키텍처/품질 | 참조: 설계 §7.2, §8.1~8.3 vs 구현 backend/src/routes/maruHeaders.js
- 설명: 설계는 Nexacro Dataset XML + ErrorCode(200 반환) 규약을 전제하나, 구현은 JSON 표준 응답과 HTTP Status 기반. 또한 설계 명시 엔드포인트(MH008 통계, MH009 clear-cache)가 라우트에 부재.
- 영향: FE 연동 실패/계약 불일치로 E2E 테스트 불가, 운영 시 예외 처리 혼선.
- 개선안:
  - 선택 A: 설계를 구현에 맞춰 JSON 표준으로 업데이트(콘텐츠타입, 에러 규약, 스웨거 일치, 응답 스키마 정의) + 통계/캐시 무효화 라우트 추가.
  - 선택 B: 구현을 설계대로 XML 계약으로 정렬(Serializer/Content-Type, ErrorCode 정책 적용) + 통계/캐시 무효화 구현.
  - 공통: 스웨거 문서에 실제 계약 반영, E2E 스펙 업데이트.

2) [DB 엔진 가정 불일치(Oracle 중심 서술 vs 기본 SQLite 운영)]
- 심각도: Medium | 카테고리: 아키텍처/품질 | 참조: 설계 §3 가정, 리포 가이드(Environment)
- 설명: 설계는 Oracle TIMESTAMP/집계/성능 특성을 가정. 저장소 기본은 SQLite(backend/data/maru.db). knex로 추상화하나 날짜/함수/타임존 처리 차이가 존재.
- 영향: 날짜 필터/성능/임계값 시험에서 DB별 상이 결과, 테스트 재현성 저하.
- 개선안: DB 추상화 레이어에 Dialect 주입(ENV로 선택), 날짜 변환/포맷을 공통 Util로 통일, 각 Dialect별 쿼리 차이 테스트 추가.

### 🟡 P3 - 보통 우선순위
3) [인증/인가 부재에 대한 완충 설계 미흡]
- 심각도: High | 카테고리: 보안 | 참조: §3 제약, §10 보안
- 설명: PoC 제약으로 미구현이나, 캐시 무효화/관리성 API는 최소 토큰 기반 보호 필요.
- 개선안: .env 기반 API Key 또는 JWT(읽기 공개, 변경 보호) 가드 레이어 설계 추가, 로깅에 권한 실패 이벤트 포함.

4) [페이징 상한치 과대(최대 1000건) 및 구현 상한 불일치]
- 심각도: Medium | 카테고리: 성능/품질 | 참조: 설계 §2(NFR-002) vs 구현 Swagger(최대 100)
- 설명: 설계 1000건은 서버/네트워크 부담. 구현 문서엔 100으로 기재되어 상이.
- 개선안: 상한 100~200로 통일, 대량 조회는 Export 전용 API/비동기 작업으로 분리.

5) [날짜/타임존 경계 및 포함 규칙 명시 필요]
- 심각도: Medium | 카테고리: 품질 | 참조: §7.1 필터, §5.1 흐름
- 설명: from/to 경계(포함/제외), 로컬/UTC 변환, 일 경계 시각 처리 미명시.
- 개선안: ISO8601(UTC) 강제, 서버 수신 즉시 UTC 정규화, [from 00:00:00, to 23:59:59.999] 포함 규칙 명시, 테스트 케이스 추가.

### 🟢 P4 - 개선 항목
6) [캐시 무효화 트리거 구체화]
- 심각도: Medium | 카테고리: 성능/품질 | 참조: §5 캐시 무효화
- 설명: "생성/수정/삭제 시 호출" 수준으로 서술. 트랜잭션 커밋 이후 후행 실행, 키 네이밍/범위(멀티 테넌시 대비) 정의 필요.
- 개선안: 서비스 레이어 공통 훅(afterCommit)에서 캐시 키 패턴 무효화, 단위 테스트로 일관성 보장.

7) [에러코드/HTTP 상태 표준화 가이드]
- 심각도: Low | 카테고리: 품질 | 참조: §8 오류 응답
- 설명: Nexacro 호환을 위해 200+ErrorCode를 택할지, 표준 HTTP Status를 택할지 프로젝트 표준 확정 필요.
- 개선안: 표준 선택과 매핑표(HTTP↔ErrorCode) 확정, 스웨거/미들웨어 공통화.

## 🎯 종합 평가
- 전반 평가: 요구사항-설계-테스트 연계가 탄탄하고 성능/운영 고려가 우수. 다만 구현체와의 계약 불일치(XML vs JSON, 엔드포인트 누락)가 현 시점 가장 큰 리스크이며, DB 엔진 가정 차이로 재현성과 성능 평가의 일관성이 흔들릴 수 있음.
- 구현 준비도: 중상. 계약 표준화와 라우트 보강 후 즉시 개발/통합 가능.
- 주요 리스크: API 계약 불일치, 인증/인가 부재, 날짜/타임존 경계 처리.
