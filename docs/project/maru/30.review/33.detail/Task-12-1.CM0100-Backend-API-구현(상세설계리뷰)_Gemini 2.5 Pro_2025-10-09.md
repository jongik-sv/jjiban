## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[이슈 제목] API 응답 형식 불일치: JSON vs XML**
- **심각도**: Critical
- **카테고리**: 아키텍처
- **설명**: 상세설계서(Task-12-1) 전반에 걸쳐 모든 API 응답이 **JSON 형식**으로 설계되었습니다(§7.2, §8). 하지만, 프로젝트의 핵심 설계 기준인 `3. api-design.md`와 `Nexacro_backend_response_sample.md`에서는 Nexacro Frontend와의 통신을 위해 **XML Dataset 형식**을 사용하도록 명확히 규정하고 있습니다. 이는 Frontend와 Backend 간의 인터페이스 계약에 대한 심각한 불일치입니다.
- **영향**: 현재 상세설계서대로 API를 구현할 경우, Nexacro Frontend는 응답을 파싱할 수 없어 전체 대시보드 기능이 동작하지 않게 됩니다. 이는 시스템의 핵심 기능 장애로 이어지는 중대한 결함입니다.
- **개선안**:
  1. 프로젝트의 표준 응답 형식을 JSON 또는 XML로 명확히 하나로 통일해야 합니다.
  2. Nexacro와의 호환성을 위해 XML Dataset 형식을 유지해야 한다면, 상세설계서의 모든 API 응답(§7.2, §8)을 `3. api-design.md`에 정의된 XML 형식으로 재설계해야 합니다.
  3. 만약 JSON으로 표준을 변경하기로 했다면, `3. api-design.md` 베이스라인 문서를 먼저 개정하고 모든 팀에 전파해야 합니다.

### 🟡 P3 - 보통 우선순위
**[이슈 제목] API 오류 처리 방식 불일치**
- **심각도**: High
- **카테고리**: 아키텍처
- **설명**: 상세설계서(Task-12-1)는 오류 발생 시 `400 Bad Request`, `503 Service Unavailable` 등 표준 HTTP 상태 코드를 사용하도록 설계되었습니다(§8.1, §8.2). 그러나 `3. api-design.md`에서는 Nexacro 통신 시 모든 응답을 **HTTP 200 OK**로 보내고, 페이로드 내의 `<ErrorCode>` 값을 통해 성공(0)과 실패(음수)를 구분하도록 정의하고 있습니다.
- **영향**: API 응답 형식 불일치 문제와 마찬가지로, Frontend는 예상치 못한 HTTP 에러 코드(4xx, 5xx)를 정상적으로 처리하지 못하고 네트워크 오류로 인식할 가능성이 높습니다.
- **개선안**: API 응답 형식을 XML로 수정하면서, 오류 처리 방식 또한 `3. api-design.md` 가이드에 맞춰 HTTP 200 상태 코드와 내부 ErrorCode를 사용하는 방식으로 일관성 있게 수정해야 합니다.

**[이슈 제목] 통계 API의 잠재적 성능 병목**
- **심각도**: Medium
- **카테고리**: 성능
- **설명**: 대시보드 통계 API(`GET /api/v1/dashboard/statistics`)는 캐시 미스 시 여러 테이블에 대해 다수의 집계(COUNT) 쿼리를 실행할 가능성이 있습니다. 데이터가 많아질 경우, 이 초기 쿼리가 NFR(비기능 요구사항)으로 정의된 2초 응답 시간을 초과할 위험이 있습니다.
- **영향**: 사용자가 처음 대시보드에 접근하거나 캐시가 만료된 후 접근할 때 응답 지연으로 인해 나쁜 사용자 경험을 유발할 수 있습니다.
- **개선안**:
  - 상세설계에 통계 쿼리 최적화 전략을 추가해야 합니다.
  - 예: 여러 COUNT 쿼리를 하나의 최적화된 쿼리로 통합하는 방안을 고려하거나, 야간 배치를 통해 미리 통계 데이터를 계산해두는 요약 테이블(Summary Table) 전략을 검토할 것을 권장합니다.

### 🟢 P4 - 개선 항목
**[이슈 제목] API Rate Limiting 적용 고려**
- **심각도**: Low
- **카테고리**: 보안/성능
- **설명**: 상세설계서 §10.1에서 API Rate Limiting을 향후 계획으로 언급하고 있으나, 대시보드 API는 사용자의 잦은 새로고침 등으로 인해 단시간에 많은 요청이 발생할 수 있습니다.
- **영향**: 악의적인 요청이나 사용자의 잦은 요청으로 인해 서버에 불필요한 부하가 발생할 수 있습니다.
- **개선안**: PoC 단계일지라도 `express-rate-limit` 같은 간단한 라이브러리를 사용하여 기본적인 요청 제한(예: 분당 60회)을 구현하는 것을 권장합니다. 이는 시스템 안정성을 초기부터 확보하는 좋은 습관입니다.

## 🎯 종합 평가
- **전반적인 설계 품질 평가**: 설계서는 매우 체계적이고 상세하여 구현에 필요한 대부분의 정보를 담고 있는 **우수한 문서**입니다. 특히 테스트 전략과 안정성 고려사항은 매우 훌륭합니다.
- **구현 준비도 평가**: **"구현 불가"** 상태입니다. P1 등급의 API 응답 형식 불일치 이슈는 Frontend와의 통신을 불가능하게 만드는 중대한 결함이므로, 해결 전에는 구현을 진행할 수 없습니다.
- **주요 리스크 및 우려사항**: 프로젝트 내 여러 설계 기준 문서(`api-design.md`, `technical-requirements.md`) 간의 내용 충돌이 가장 큰 리스크입니다. 이는 다른 Task에서도 동일한 문제를 야기할 수 있으므로 조속한 표준 정립이 필요합니다.
