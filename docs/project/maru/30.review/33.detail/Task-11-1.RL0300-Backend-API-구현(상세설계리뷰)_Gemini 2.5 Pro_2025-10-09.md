### 🟠 P2 - 빠른 해결 권장
**[이슈-01] SCRIPT 연산자의 보안 위험성**
- **심각도**: High
- **카테고리**: 보안
- **설명**: SCRIPT 연산자는 JavaScript 표현식을 평가하도록 설계되었으나, PoC에서는 단순 문자열 비교로 제한한다고 명시되어 있습니다. 하지만 향후 `vm2` 또는 `isolated-vm` 같은 샌드박스 환경을 사용하더라도, 외부 입력 기반의 동적 코드 실행은 잠재적인 코드 인젝션(Code Injection) 공격에 노출될 위험이 매우 큽니다.
- **영향**: 악의적인 스크립트가 서버 컨텍스트에서 실행될 경우, 시스템 장애, 데이터 유출, 또는 서버 장악까지 이어질 수 있는 심각한 보안 취약점이 될 수 있습니다.
- **개선안**:
  1. **실행 제한**: SCRIPT 연산자의 사용을 완전히 금지하거나, 사전에 정의되고 검증된 스크립트만 ID로 호출하는 화이트리스트 방식으로 변경하는 것을 강력히 권장합니다.
  2. **안전한 대안 도입**: 동적 로직이 반드시 필요하다면, JavaScript 대신 JSON Logic, JSLT 등과 같이 더 안전하고 제한된 형태의 규칙 엔진을 도입하는 것을 검토해야 합니다.
- **참조**: OWASP Top 10 - A03:2021-Injection

### 🟡 P3 - 보통 우선순위
**[이슈-02] 디버그 API의 정보 노출 가능성**
- **심각도**: Medium
- **카테고리**: 보안
- **설명**: `debug: true` 파라미터를 통해 룰 실행 시간, 평가된 룰 개수, 조건별 결과 등 상세한 내부 정보가 응답으로 반환됩니다. 이 기능이 운영 환경에서도 활성화될 경우, 시스템의 내부 로직과 성능 특성을 공격자에게 노출할 수 있습니다.
- **영향**: 공격자가 시스템의 동작 방식을 파악하여 추가 공격(예: 타이밍 공격, 로직 취약점 탐색)을 시도하는 데 악용될 수 있습니다.
- **개선안**:
  1. **환경 분리**: `debug` 파라미터는 `development` 또는 `test` 환경에서만 동작하도록 환경 변수(`NODE_ENV`)에 따라 분기 처리해야 합니다.
  2. **권한 제어**: 운영 환경에서 디버깅이 반드시 필요한 경우, 특정 관리자 역할(Role)을 가진 사용자에게만 허용하도록 접근 제어를 추가해야 합니다.
- **참조**: 상세설계서 §7.2, §8.1

**[이슈-03] BETWEEN 연산자의 데이터 타입 처리 모호성**
- **심각도**: Medium
- **카테고리**: 품질
- **설명**: BETWEEN 연산자는 `10 AND 20`과 같은 형식으로 정의되어 있습니다. 하지만 피연산자가 숫자, 문자열, 날짜 중 어떤 타입으로 비교되어야 하는지에 대한 명확한 정의가 없습니다. 예를 들어, 문자열 '5'는 '10'과 '2' 사이에 위치하게 되어 예기치 않은 결과를 초래할 수 있습니다.
- **영향**: 데이터 타입에 따라 정렬 순서가 달라져 룰 평가가 의도와 다르게 동작하고, 잘못된 결과를 반환할 수 있습니다.
- **개선안**:
  1. **타입 명시**: 룰 변수 정의(TB_MR_RULE_VAR)의 `VARIABLE_DATA_TYPE`에 따라 비교 방식을 명확히 분기해야 합니다.
  2. **구현 구체화**: 상세설계서 §5.3.1의 `convertValue` 함수 로직을 BETWEEN 연산자 평가 시에도 명시적으로 적용하도록 수정해야 합니다.
- **참조**: 상세설계서 §5.1.3, §5.3.2

### 🟢 P4 - 개선 항목
**[이슈-04] 룰 실행 API의 성능 저하 가능성**
- **심각도**: Low
- **카테고리**: 성능
- **설명**: 룰 실행 시, 매번 DB에서 모든 활성 룰 레코드를 조회하여 메모리에서 순차적으로 평가합니다. 마루(Maru)에 포함된 룰의 개수가 수백 개 이상으로 증가하면 DB 조회 및 평가 로직에서 병목이 발생할 수 있습니다.
- **영향**: 시스템 부하 증가 및 API 응답 시간 지연으로 이어질 수 있습니다.
- **개선안**:
  1. **캐싱 전략 구체화**: 상세설계서 §11.3에 언급된 '룰 레코드 캐싱'을 구체화하여, 자주 사용되는 마루의 룰 레코드를 메모리(예: Redis)에 캐시하는 전략을 초기 설계에 포함하는 것을 권장합니다.
  2. **인덱싱된 조건 평가**: 자주 사용되는 조건 변수(OP_1, OP_2 등)에 대해 부분적인 인덱스를 생성하여 DB 조회 시점에서 1차 필터링을 수행하는 방안을 고려할 수 있습니다.
- **참조**: 상세설계서 §11.2

## 🎯 종합 평가
- **전반적인 설계 품질**: 우수. 상세설계서가 매우 체계적이고 구체적으로 작성되어 있어 구현의 기반으로 삼기에 충분합니다. 특히 API 계약과 테스트 케이스가 상세하여 개발 효율성을 높일 것으로 기대됩니다.
- **구현 준비도**: 높음. 발견된 이슈들은 대부분 설계의 논리적 결함보다는 잠재적 위험이나 모호성을 명확히 하는 것에 초점이 맞춰져 있습니다. P2, P3 이슈 해결 후 즉시 구현 착수가 가능합니다.
- **주요 리스크**: `SCRIPT` 연산자 관련 보안 위험(이슈-01)이 가장 큰 우려 사항입니다. PoC 범위를 넘어 실제 운영 환경에 적용하기 전 반드시 해결되어야 할 문제입니다.
