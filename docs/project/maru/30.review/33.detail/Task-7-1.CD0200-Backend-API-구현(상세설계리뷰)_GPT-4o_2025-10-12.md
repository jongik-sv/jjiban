## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[ALTER_CODE_NAME 다중 별칭의 유일성·동기화 누락]**
- **심각도**: High · **카테고리**: 데이터 무결성/아키텍처
- **설명**: ALTER_CODE_NAME1~5 컬럼은 동일 Code Base에 대한 다국어/별칭으로 사용되지만, 문서에 유니크 제약·중복 검사·정규화 방안이 없다. 다국어 번역 Sync도 정의되지 않아 동일 별칭이 다수 코드에 중복될 수 있다.
- **영향**: 검색·자동완성·보고서 등에서 다의적 매핑/충돌, 추후 삭제·수정 시 연쇄 오류 발생.
- **개선안**: (1) 별칭 컬럼에 대해 Unique Index(LOWER 값 기준) 또는 별도 별칭 테이블 도입, (2) 입력 시 기존 별칭 검색·경고, (3) 번역 메타데이터/동기화 전략을 명시하고 Swagger/Joi 규칙에 반영.

### 🟠 P2 - 빠른 해결 권장
**[정렬 재배치(SORT_ORDER) 동시성·트랜잭션 불명확]**
- 병렬 API 호출 시 순위가 엇갈릴 수 있으나, 재배치 알고리즘·락 전략·배치 업데이트 로직이 상세히 기술되지 않음. → 순위 재정렬용 단일 트랜잭션/Version 컬럼/배치 업데이트 규칙 명시 필요.

**[CB001 목록 조회의 페이징/정렬 계약 부족]**
- page/limit/sort 파라미터 범위·기본값·허용 필드·오류코드가 산발적으로만 언급. Swagger 공통 스키마/400 응답 예시, 클라이언트 Validation 기준을 보강 필요.

**[캐싱/조회 성능 지표 구체화 필요]**
- 최대 데이터량·인덱스 구성·캐시 무효화/TTL 전략이 “향후 확장” 수준으로만 기재. → 주요 쿼리별 Index 권장(예: (CATEGORY_ID,SORT_ORDER), (MARU_ID,START/END_DATE))과 캐시 키/TTL/무효화 트리거 문서화 권장.

### 🟡 P3 - 보통 우선순위
- **에러 모델**: ErrorCode/Message/추적ID 구조와 HTTP Status 매핑을 표준화해 Swagger에 예시 추가 필요.
- **삭제 정책**: Code Base 삭제 시 Child 종속성/기간 버전 처리 전략이 명확치 않음 (Soft delete, Status 변경 등).
- **감사/로깅**: CRUD·정렬 변경 이력 로깅/모니터링 지표(요청 수·실패율·평균 응답시간) 정의 필요.

### 🟢 P4 - 개선 항목
- 다국어/국제화: ALTER_CODE_NAME을 i18n 자원으로 분리하여 프런트와 공유하는 전략 마련.
- 보안: Rate limit, ID/별칭 길이/패턴 제한, 관리자 권한 모델 명시.

### 🔵 P5 - 참고 사항
- 향후 Redis 등 분산 캐시 도입 시 멀티 인스턴스 무효화 전략 사전 정의 권장.
