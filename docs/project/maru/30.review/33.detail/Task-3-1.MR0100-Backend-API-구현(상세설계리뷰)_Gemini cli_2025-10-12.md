### 🟡 P3 - 보통 우선순위
**[이슈-01] 비표준 REST API 응답 규격**
- **심각도**: Medium
- **카테고리**: 🏗️ 아키텍처
- **설명**: `api-design.md`에 따르면, Nexacro 플랫폼과의 연동을 위해 모든 Backend 응답이 HTTP 상태 코드 `200 OK`로 고정되어 있고, 실제 성공/실패 여부는 응답 본문의 `<ErrorCode>` 값으로 판단합니다. 이는 RESTful API의 표준적인 에러 처리 방식(4xx, 5xx 상태 코드 사용)과 다릅니다.
- **영향**:
  - REST API 표준을 따르는 다른 클라이언트(모바일 앱, 웹 등)와의 통합 시 혼란을 유발할 수 있습니다.
  - HTTP 인프라(캐싱, 모니터링, 게이트웨이 등)가 응답을 모두 성공으로 오인하여 에러 추적 및 분석이 어렵습니다.
- **개선안**: Nexacro 플랫폼의 제약사항이 아니라면, REST API 표준에 따라 클라이언트 에러는 4xx, 서버 에러는 5xx 상태 코드를 사용하도록 응답 규격을 재검토할 것을 권장합니다. 만약 제약사항이라면, 해당 사유를 API 문서에 명확히 기재해야 합니다.
- **참조**: `docs/project/maru/00.foundation/02.design-baseline/3. api-design.md` - 섹션 3.4

### 🟢 P4 - 개선 항목
**[이슈-02] 낙관적 락(Optimistic Lock) 구현 방안 구체화 필요**
- **심각도**: Low
- **카테고리**: 📝 품질
- **설명**: 상세설계서의 "동시성 제어" 부분에서 `VERSION` 컬럼을 이용한 낙관적 락을 언급했으나, 실제 `UPDATE` 쿼리에서 버전을 어떻게 확인할 것인지에 대한 구체적인 구현 방안이 명시되어 있지 않습니다.
- **영향**: 동시 수정 상황에서 데이터 정합성이 깨질 수 있는 잠재적 위험이 있습니다.
- **개선안**: 데이터 수정(`PUT`, `PATCH`) 시, `WHERE` 절에 `maruId`와 함께 `version`을 포함하여 동시성 충돌을 감지하는 로직을 명시해야 합니다.
  ```sql
  -- 예시
  UPDATE TB_MR_HEAD
  SET maruName = :newName, ...
  WHERE maruId = :maruId AND version = :expectedVersion;
  ```
- **참조**: `Task-3-1.MR0100-Backend-API-구현(상세설계).md` - 섹션 9.2, 11.2

**[이슈-03] 논리적 삭제를 위한 HTTP Verb 재고려**
- **심각도**: Low
- **카테고리**: 🏗️ 아키텍처
- **설명**: 리소스를 물리적으로 삭제하지 않고 `END_DATE`만 갱신하는 논리적 삭제(Soft Delete) 기능에 `DELETE` HTTP 메서드를 사용하고 있습니다.
- **영향**: REST API의 의미론적 일관성이 다소 저하됩니다. `DELETE`는 리소스의 완전한 제거를 기대하게 만듭니다.
- **개선안**: 이는 논쟁의 여지가 있는 부분이지만, 리소스의 상태를 변경하는 관점에서 `PATCH` 메서드(예: `PATCH /maru-headers/{id} {"status": "DELETED"}`)를 사용하는 것을 고려해볼 수 있습니다. 단, 현재 설계도 널리 사용되는 방식이므로 팀의 컨벤션을 따르는 것이 중요합니다.
- **참조**: `Task-3-1.MR0100-Backend-API-구현(상세설계).md` - API MH005

**[이슈-05] 인증/권한 관리 부재**
- **심각도**: Info
- **카테고리**: 🛡️ 보안
- **설명**: 상세설계서에 명시된 바와 같이, PoC 범위에서 인증 및 권한 관리(Authentication/Authorization) 기능이 제외되었습니다.
- **영향**: 현재 API는 보호되지 않으므로, 프로덕션 환경에 배포될 수 없습니다.
- **개선안**: PoC 이후 정식 버전 개발 착수 시, JWT 기반 인증 및 역할 기반 접근 제어(RBAC)를 최우선으로 구현해야 합니다. 이를 기술 부채(Technical Debt) 항목으로 관리할 것을 권장합니다.
- **참조**: `Task-3-1.MR0100-Backend-API-구현(상세설계).md` - 섹션 3.3

## 🎯 종합 평가
- **전반적인 설계 품질**: **우수**. 요구사항이 명확하고, API 계약이 상세하며, 테스트 전략이 구체적이어서 구현 단계의 불확실성을 크게 줄였습니다. 특히 선분 이력 모델에 대한 깊은 고민이 돋보입니다.

- **주요 리스크**: 기본설계서 부재로 인한 잠재적 요구사항 누락 가능성이 유일한 리스크입니다.
