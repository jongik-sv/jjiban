## ⚠️ 발견된 이슈

### 🟠 P2 - 빠른 해결 권장
**[이슈] 인증/인가 부재에 따른 보안 위험**
- **심각도**: High
- **카테고리**: 🛡️ 보안
- **설명**: 설계서에 "PoC 단계로 인증/권한 관리가 구현되지 않음"이라고 명시되어 있습니다. 이는 PoC 단계에서는 수용 가능한 제약일 수 있으나, 운영 시스템에서는 심각한 보안 취약점이 됩니다.
- **영향**: 인가되지 않은 사용자가 API에 접근하여 데이터를 조회, 조작하거나 시스템에 부하를 유발할 수 있습니다.
- **개선안**: 정식 버전 전환 시, JWT(JSON Web Token) 기반의 인증 및 역할 기반의 권한 관리(RBAC)를 도입하는 설계를 추가해야 합니다. API Gateway 등을 활용하여 인증/인가를 중앙에서 처리하는 방안을 고려할 수 있습니다.

### 🟡 P3 - 보통 우선순위
**[이슈] 캐시 무효화(Invalidation) 트리거 불명확**
- **심각도**: Medium
- **카테고리**: 🏗️ 아키텍처
- **설명**: 통계 정보 캐시를 무효화하기 위한 `POST /api/v1/maru-headers/clear-cache` API가 설계되어 있습니다. 문서에는 "마루 헤더 생성/수정/삭제 시 Service에서 캐시 클리어 호출"이라고 되어 있으나, 해당 C/U/D 기능은 본 설계의 범위(Scope)에서 제외되어 있습니다.
- **영향**: 캐시 무효화 로직이 실제로 어떻게 호출되는지 불분명하여, 데이터 변경 후에도 통계 정보가 갱신되지 않아 데이터 불일치가 발생할 수 있습니다.
- **개선안**: 본 설계 문서에서 C/U/D 기능을 구현하는 다른 Task(예: Task 3.1, 5.1)와의 명시적인 인터페이스 계약을 정의하거나, 이벤트 기반 아키텍처(예: 데이터 변경 이벤트 발행 시 캐시 무효화 구독자가 처리) 도입을 고려하여 결합도를 낮추는 방안을 제시해야 합니다.

### 🟢 P4 - 개선 항목
**[이슈] API 오류 응답 시 HTTP 상태 코드 비일관성**
- **심각도**: Low
- **카테고리**: 📝 품질
- **설명**: 입력 파라미터 유효성 검증 실패 등 클라이언트 측 오류에 대해 HTTP 상태 코드 `200 OK`와 내부 `ErrorCode`를 조합하여 응답하도록 설계되었습니다. 이는 Nexacro 플랫폼의 제약일 수 있으나, 현대적인 REST API 설계 원칙에는 부합하지 않습니다.
- **영향**: API 클라이언트가 HTTP 표준만으로는 요청의 성공/실패를 직관적으로 판단하기 어려우며, 모니터링 시스템에서 오류율을 집계할 때 혼선을 줄 수 있습니다.
- **개선안**: 기술적인 제약이 없다면, 클라이언트 오류(잘못된 요청)는 `400 Bad Request`, 서버 오류는 `500 Internal Server Error` 등 표준 HTTP 상태 코드를 반환하도록 설계를 변경하는 것을 권장합니다. Nexacro 클라이언트가 표준 상태 코드 처리가 가능하다면, 이를 따르는 것이 장기적으로 유지보수성에 유리합니다.

**[이슈] 숫자(Number) 타입 데이터의 문자열(String) 변환**
- **심각도**: Low
- **카테고리**: 📝 품질
- **설명**: §7.3에서 "숫자 타입 → 문자열 변환 (Nexacro Dataset 호환성)"이라고 명시되어 있습니다. 이 역시 플랫폼 제약일 수 있으나, 데이터의 고유한 타입 정보가 손실되는 문제입니다.
- **영향**: Frontend에서 해당 값을 이용해 숫자 연산, 정렬, 포맷팅 등을 수행할 때 불필요한 타입 변환이 필요하거나 오류가 발생할 수 있습니다.
- **개선안**: Nexacro 플랫폼에서 숫자 타입을 직접 처리할 수 있는지 재확인하고, 가능하다면 원본 데이터 타입을 유지하여 전달하는 것을 권장합니다. 이것이 불가능하다면, 해당 제약사항과 Frontend에서 처리 시 주의사항을 명확하게 문서화해야 합니다.

### 🔵 P5 - 참고 사항
**[이슈] 기본설계서 부재로 인한 요구사항 추적성 검증 한계**
- **심각도**: Info
- **카테고리**: 📝 품질
- **설명**: 상세설계의 기반이 되는 `Task-4-1`의 기본설계서가 발견되지 않았습니다. 이로 인해 상세설계가 기본설계의 모든 요구사항을 충족하는지에 대한 완전한 추적성 검증을 수행할 수 없었습니다.
- **영향**: 기본설계의 중요 요구사항이 누락되었을 가능성을 배제할 수 없습니다.
- **개선안**: 프로젝트 관리 프로세스에 따라 기본설계서 작성을 완료하고, 해당 문서를 기반으로 상세설계의 완전성을 재검증해야 합니다.
