## ✅ 적용 대상 상세 (9개)

### [1] 중복 요청 방지 미비 (GPT-4o P1)
- **원본**: CUD 액션 더블클릭/연속 클릭 방지, in-flight 요청 취소 정책 미비
- **방식**: 📝 조정 적용
- **내용**:
  - 버튼 클릭 시 즉시 비활성화 (Nexacro Button.enable = false)
  - Transaction 완료 시 재활성화
  - idempotency key는 프로덕션 대비 (PoC 제외)
- **섹션**: §5 프로세스 흐름, §9 오류 처리
- **이유**: PoC 단일 사용자 환경이나 UX 품질 향상, Nexacro 기본 기능으로 간단 구현 가능

### [2] Dataset END_DATE 필드 누락 (Qwen P1)
- **원본**: ds_maruList에는 있으나 ds_maruDetail에 누락
- **방식**: ✅ 그대로 적용
- **내용**: ds_maruDetail 컬럼 정의에 END_DATE 추가
- **섹션**: §7.1 Dataset 정의
- **이유**: 데이터 정합성 필수, 화면 표시 필요

### [3] 입력 검증/정규화 명세 세분화 (GPT-4o P2)
- **원본**: 필드별 길이/패턴/허용값 표준 표로 명시 필요
- **방식**: ✅ 그대로 적용
- **내용**:
  - MARU_ID: 패턴 `^[A-Za-z0-9_]{1,20}$`, 최대 20자
  - MARU_NAME: 1-100자, 특수문자 제한
  - MARU_TYPE: 열거형 "CODE"|"RULE"
  - 날짜 필드: YYYYMMDDHHmmss 형식
- **섹션**: §7.4 유효성 검증 규칙 (표 보완)
- **이유**: 개발자 구현 가이드 명확화, 백엔드 Joi 스키마와 동기화

### [4] 에러 모델/상태코드 매핑 (GPT-4o P2)
- **원본**: 서버 에러를 UI 메시지로 변환하는 규칙 정형화
- **방식**: ✅ 그대로 적용
- **내용**:
  - ErrorCode → 사용자 메시지 매핑 표
  - 필드 단위 vs 글로벌 오류 구분
  - 포커스 이동 정책
  - 재시도/취소 가이드
- **섹션**: §9 오류 처리 (매핑 표 추가)
- **이유**: 일관된 UX, 테스트 케이스와 연동

### [5] 상태 관리 로직 명확화 (Qwen P2)
- **원본**: INUSE 상태 수정 시 선분 이력 생성 로직 불명확
- **방식**: ✅ 그대로 적용
- **내용**:
  - CREATED 상태: 직접 수정 (VERSION 유지)
  - INUSE 상태:
    1. 기존 행 END_DATE를 현재 시각으로 갱신
    2. 새 행 생성 (VERSION++, START_DATE=현재, END_DATE=9999)
  - DEPRECATED 상태: 수정 불가 (버튼 비활성화)
- **섹션**: §5.1 프로세스 설명 (기존 마루 수정 단계 보완)
- **이유**: 선분 이력 모델 핵심 로직, 개발자 이해 필수

### [6] 환경설정/엔드포인트 관리 (GPT-4o P3)
- **원본**: baseURL, timeout, 재시도 등 환경 설정 분리
- **방식**: ✅ 그대로 적용
- **내용**:
  - Nexacro 전역 변수로 설정 정의
  - `application.gv_baseURL = "http://localhost:3000"`
  - `application.gv_timeout = 30000`
  - 문서에 설정 항목 표 추가
- **섹션**: §4.2 외부 의존성 (설정 관리 항목 추가)
- **이유**: 환경별 배포 용이, 모범 사례

### [7] 접근성 체크리스트 정교화 (GPT-4o P4)
- **원본**: Tab 순서, 포커스 링, aria-label 강화
- **방식**: ✅ 그대로 적용
- **내용**:
  - 포커스 순서 명시: 검색영역 → 액션버튼 → 그리드 → 상세정보
  - 주요 필드 aria-label 추가
  - aria-invalid 적용 (유효성 검증 실패 시)
  - 키보드 단축키 일관성
- **섹션**: §6.3 접근성 가이드 (체크리스트 보완)
- **이유**: 웹 표준 준수, Nexacro 기본 지원 활용

### [8] API 응답 처리 일관성 (Qwen P4)
- **원본**: 공통 Transaction 처리 패턴 정의
- **방식**: ✅ 그대로 적용
- **내용**:
  - 공통 callback 함수 패턴 정의
  - ErrorCode 0: 성공 처리 (메시지 + 목록 갱신)
  - ErrorCode ≠ 0: 오류 처리 (ErrorMsg 표시)
  - 네트워크 오류: 재시도 로직
- **섹션**: §8 인터페이스 계약 (공통 처리 패턴 추가)
- **이유**: 코드 중복 제거, 유지보수성 향상

### [9] 성능 측정 환경 명시 (Qwen P4)
- **원본**: 테스트 환경 (하드웨어, 네트워크, 데이터셋) 명확화
- **방식**: ✅ 그대로 적용
- **내용**:
  - 테스트 환경: Desktop (1920x1080), Chrome 최신
  - 네트워크: Localhost (PoC), 3G 시뮬레이션 (프로덕션)
  - 데이터셋: 50건 (PoC), 1000건 (성능 테스트)
- **섹션**: §11 성능 및 확장성 (측정 환경 표 추가)
- **이유**: 성능 검증 객관성 확보
