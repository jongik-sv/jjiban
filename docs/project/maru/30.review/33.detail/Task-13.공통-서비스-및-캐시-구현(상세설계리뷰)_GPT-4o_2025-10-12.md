## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[단일 인스턴스 node-cache 사용에 대한 멀티 노드 전략 부재]**
- **심각도**: High · **카테고리**: 아키텍처/운영
- **설명**: @cacheable/node-cache(프로세스 메모리)를 사용하면서 멀티 인스턴스/수평 확장 시 캐시 동기화, 무효화, Warm-up 전략이 전혀 없다.
- **영향**: 다중 서버 운영 시 데이터 불일치, 오래된 정보 노출, 캐시 무효화 실패 → 시스템 일관성 붕괴.
- **개선안**: (1) Redis 등 분산 캐시 전환 또는 Pub/Sub 기반 캐시 동기화, (2) 캐시 키/TTL/무효화 이벤트 정의, (3) 단일 인스턴스 제한/배포 절차 문서화, (4) 캐시 미사용 Fallback 정책.

### 🟠 P2 - 빠른 해결 권장
**[캐시 무효화 이벤트·옵저버 설계 부족]**
- CRUD/상태 전이 발생 시 어떤 이벤트가 어떤 캐시를 비우는지 명시되어 있지 않음. → API별 무효화 포인트 정의 필요.

**[Validation/에러 핸들링 모듈의 스키마 정의 미흡]**
- Joi 스키마/에러 코드 매핑/로깅 형식이 추상적. → 표준화 및 Swagger/Joi/로깅 연동 필요.

**[비동기 작업/Retry 정책]**
- 공통 서비스에서 외부 호출이나 DB 에러 발생 시 재시도/백오프/알림 전략 미흡.

### 🟡 P3 - 보통 우선순위
- **로깅/모니터링**: Structured Log 필드, 상관관계 ID, Alert 룰 표준화 필요.
- **테스트 전략**: 공통 함수별 단위 테스트/통합 테스트 커버리지 목표·도구 명시.
- **보안**: 캐시 내 민감 정보 저장 여부와 암호화/마스킹 정책 부재.

### 🟢 P4 - 개선 항목
- 캐시 미스/히트·Invalidate/에러률 모니터링 지표를 수집해 대시보드화.
- Blue/Green 배포 시 캐시 Warm-up/Drain 절차 제시.

### 🔵 P5 - 참고 사항
- 추후 Observability 플랫폼 연계 시 로깅/메트릭 표준을 Prometheus/Grafana와 정합 확보 필요.
