## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[SCRIPT 실행 보안·샌드박스 전략 부재]**
- **심각도**: Critical · **카테고리**: 보안/아키텍처
- **설명**: SCRIPT 유형을 허용하지만 실행 엔진, 샌드박스, 타임아웃, 허용 API/객체 제한이 전혀 정의돼 있지 않음.
- **영향**: 악성 스크립트 실행 시 시스템 제어권 탈취, 무한 루프·리소스 고갈, 데이터 유출 등 치명적 리스크.
- **개선안**: (1) SCRIPT 사용을 제한하거나 승인 워크플로 도입, (2) 별도 샌드박스 프로세스/VM + CPU·메모리·시간 제한, (3) 허용 함수/객체 화이트리스트, (4) 실행 로그·감사·알림 체계 마련. Swagger/Joi에 제한 반영.

### 🟠 P2 - 빠른 해결 권장
**[실행 SLA(500ms) 대비 캐시/프리컴파일 전략 부재]**
- 실행마다 Rule Record/Variable 조합을 계산하므로 캐시 키/TTL, Pre-compiled 파서 전략 필요.

**[동시 실행/락 정책 미흡]**
- 동일 Rule에 대한 동시 실행·버전 전환 시점에서 Race Condition 처리(읽기 스냅샷/버전 고정)가 정의되지 않음.

**[에러 모델/코드 표준화 부족]**
- ErrorCode와 HTTP Status, 메시지 템플릿(사용자/개발자용) 분리가 미흡. SCRIPT 실패, 연산자 미지원, 데이터 누락 등 사례별 메시지 필요.

### 🟡 P3 - 보통 우선순위
- **로깅/모니터링**: 실행 시작/종료/결과(성공·실패·경고), 소요시간, 입력 요약 등을 구조화 로그/지표로 정의해야 함.
- **보안**: 입력 길이/허용 문자/JSON 크기 제한, Rate Limit 정책 추가 필요.
- **테스트**: ReDos 패턴, 경계 연산 (=, BETWEEN, IN), SCRIPT 실패 테스트 자동화 항목 보강.

### 🟢 P4 - 개선 항목
- Rule Simulator API 제공(샌드박스에서 미리 실행) 검토.
- SLA 위반 시 Graceful Degradation(타임아웃 반환/부분 실행) 설계.

### 🔵 P5 - 참고 사항
- 향후 실시간 모니터링/알림(Apm, Metric Export) 계획을 수립하면 운영 대응력 향상.
