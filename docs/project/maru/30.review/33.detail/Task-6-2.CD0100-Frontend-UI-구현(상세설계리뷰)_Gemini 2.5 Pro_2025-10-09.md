### 🟡 P3 - 보통 우선순위
**[이슈 제목] RegEx 실시간 검증 시 UI 프리징 위험**
- **심각도**: Medium
- **카테고리**: 성능/품질
- **설명**: '코드 정의' 입력란에 RegEx(정규식) 타입 선택 시, `onchanged` 이벤트를 통해 실시간 검증을 수행하도록 설계되어 있습니다. 만약 사용자가 복잡하거나 잘못된 정규식(e.g., Catastrophic Backtracking을 유발하는 패턴)을 입력할 경우, UI 스레드가 블로킹되어 애플리케이션 전체가 멈추는 현상이 발생할 수 있습니다.
- **영향**: 사용자 경험에 치명적이며, 애플리케이션을 재시작해야 할 수 있습니다.
- **개선안**:
  1.  **Asynchronous Validation**: 정규식 검증 로직을 비동기로 처리합니다.
  2.  **Web Worker 활용**: 검증 로직을 별도의 Web Worker 스레드에서 실행하여 UI 스레드의 블로킹을 원천적으로 방지합니다.
  3.  **Timeout 설정**: 검증 실행 시간을 200ms 등으로 제한하고, 시간 초과 시 "검증 시간 초과: 정규식이 너무 복잡합니다."와 같은 안내 메시지를 표시하는 방어 코드를 추가하는 것을 권장합니다.

**[이슈 제목] 삭제 제약조건 발생 시 사용자 경험 저하**
- **심각도**: Medium
- **카테고리**: 품질/사용성
- **설명**: 카테고리 삭제 시, 해당 카테고리를 사용하는 코드 기본값이 존재하면 API는 409 Conflict 에러를 반환하도록 되어 있습니다. 현재 설계는 이 에러 발생 시 "관련 코드가 존재하여 삭제할 수 없습니다." 라는 메시지만 표시합니다. 사용자는 어떤 코드가 문제인지 알 수 없어 직접 찾아야 하는 불편함이 있습니다.
- **영향**: 사용자가 문제 해결을 위해 불필요한 탐색 과정을 거쳐야 하므로 작업 효율성이 저하됩니다.
- **개선안**:
  1.  **Backend API 개선**: 삭제 API(CC005)의 409 에러 응답 본문에 삭제를 방해하는 코드 목록(예: 최대 5개)을 포함하도록 개선을 요청합니다.
  2.  **Frontend 개선**: 에러 메시지 다이얼로그에 "삭제를 방해하는 코드: [코드1, 코드2, ...]"와 같이 상세 정보를 함께 표시하여 사용자가 문제를 즉시 인지하고 조치할 수 있도록 UI를 개선합니다.

### 🟢 P4 - 개선 항목
**[이슈 제목] XSS 방어 전략 구체화 필요**
- **심각도**: Low
- **카테고리**: 보안
- **설명**: 상세설계서 10.1절에 XSS 방지 전략으로 "HTML 태그 입력 차단"이라고 명시되어 있습니다. 이는 좋은 방향이지만, 구체적인 구현 방법이 없어 개발자마다 다르게 해석할 여지가 있습니다.
- **영향**: 일관되지 않은 보안 처리로 인해 잠재적인 X.S.S 취약점이 발생할 수 있습니다.
- **개선안**: "Nexacro의 `nexacro.getText()` 함수를 사용하여 HTML 특수문자를 이스케이프 처리" 또는 "서버 사이드에서 `DOMPurify`와 같은 검증된 라이브러리를 사용하여 화이트리스트 기반으로 HTML을 새니타이즈" 와 같이 구체적인 처리 방안을 명시하여 일관성을 확보할 것을 권장합니다.

**[이슈 제목] UI 설계 SVG 파일 누락**
- **심각도**: Low
- **카테고리**: 품질/문서화
- **설명**: 상세설계서 6.2절에 UI 설계 SVG 파일(`Task-6-2.CD0100-Frontend-UI-구현_UI설계.svg`)이 참조되어 있으나, 실제 파일은 생성되지 않았습니다.
- **영향**: 텍스트 기반 레이아웃만으로는 개발자가 UI의 시각적 구조와 상호작용을 직관적으로 파악하기 어렵습니다.
- **개선안**: 텍스트 레이아웃을 기반으로 실제 SVG 파일을 생성하여 문서에 포함해주세요.
