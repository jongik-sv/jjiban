- 기간 기반(start/end)과 상태 규칙을 설명하며 Nexacro Dataset 변환, Swagger 문서화 계획 포함.
- Joi 기반 입력 검증, node-cache 활용 등 품질/성능 보강 장치 언급.

## ⚠️ 발견된 이슈

### 🔴 P1 - 즉시 해결 필요
**[부모-자식 무결성 및 순환 참조 검증 부재]**
- **심각도**: High
- **카테고리**: 품질/아키텍처
- **설명**: 설계에서 Parent-Code/Child-Code 구조를 언급하지만 저장/수정 시 부모 존재 여부, 자기 참조, 순환 방지, 동일 depth 제한 등에 대한 검증 로직/규칙이 명시돼 있지 않음.
- **영향**: 잘못된 계층 저장으로 런타임 오류, 무한 순환, UI 표시 실패, 데이터 정합성 붕괴.
- **개선안**: 부모 코드 존재 확인, 자기 참조/순환 탐지(DFS 또는 DB 제약), depth 제한 정책 문서화 및 테스트 케이스 추가.

### 🟠 P2 - 빠른 해결 권장
**[RegEx 패턴 검증/안전성 기준 미흡]**
- **설명**: RegEx 유형 지원을 언급하지만 허용 패턴 길이, 재귀/역참조 제한, Catastrophic Backtracking 방지(Timeout) 정책이 없음.
- **개선안**: 허용 길이/복잡도, 시간 제한, 사전 컴파일 및 테스트, 유해 패턴 거부 규칙 수립.

**[상태 전이 동시성/트랜잭션 범위 불명확]**
- **설명**: CREATED→INUSE→DEPRECATED 규칙은 있으나 다중 트랜잭션 동시 요청, optimistic lock/version 필드, audit 로깅 전략이 누락.
- **개선안**: 버전 컬럼/updated_at 기반 낙관적 잠금, 상태 전이 로그, 중복 요청 차단 정책 정의.

**[페이징·정렬 파라미터 규격화 필요]**
- **설명**: 목록 조회 page/limit/sort/filter 구성이 산발적으로 언급, 범위 초과·음수 값 처리 미정.
- **개선안**: 공통 파라미터 계약, 기본값/최대값, 허용 정렬 필드, 오류 코드 표를 Swagger에 반영.

### 🟡 P3 - 보통 우선순위
**[Cascade 삭제/이력 정책 누락]**
- 자식 코드 존재 시 삭제 금지/Soft delete/이전 상태 보존 정책이 명확치 않음.

**[캐시 무효화/TTL 정책 미정]**
- node-cache 사용만 언급, CRUD/상태 전이/RegEx 수정 시 캐시 키 무효화 기준과 TTL 관리가 없음.

**[에러 모델/메시지 표준화 필요]**
- 코드·메시지·추적ID 구조와 HTTP 상태코드 매핑, Nexacro Dataset 에러 스키마가 문서화되지 않음.

### 🟢 P4 - 개선 항목
- 로깅/감사: CRUD/상태 전이/RegEx 수정 이벤트 로깅과 모니터링 지표 정의.
- 보안: API rate-limit, ID 길이/패턴 제약, 관리자 권한/역할 매핑 명시.

### 🔵 P5 - 참고 사항
- 향후 Redis 등 외부 캐시 사용 시 멀티 인스턴스 간 동기화 전략 고려 권장.
