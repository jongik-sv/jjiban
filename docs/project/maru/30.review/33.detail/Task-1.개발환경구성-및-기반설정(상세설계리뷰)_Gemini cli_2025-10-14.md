## ⚠️ 발견된 이슈

### 🟠 P2 - 빠른 해결 권장
**[이슈 제목] Nexacro 개발 환경 버전 불일치**
- **심각도**: High
- **카테고리**: 아키텍처
- **설명**: 상세설계서(1.2, 3.3, 9.1절 등)와 `tasks.md`에서는 "Nexacro Platform 17"을 명시하고 있으나, `technical-requirements.md`, `system-architecture.md` 등 다수의 핵심 기반설계 문서에서는 "Nexacro N v24"를 기술 스택으로 지정하고 있습니다.
- **영향**: 개발 환경 구성의 핵심적인 부분에서 버전 불일치가 발생하여, 잘못된 버전의 개발 환경이 구축될 수 있습니다. 이는 추후 UI 개발 전체에 영향을 미치는 심각한 재작업을 유발할 수 있습니다.
- **개선안**: 프로젝트의 표준 Nexacro 버전을 "Nexacro N v24"로 명확히 확정하고, 본 상세설계서를 포함한 모든 관련 문서에서 버전을 일관되게 수정해야 합니다.
- **참조**: `technical-requirements.md` (2.2절), `system-architecture.md` (2.1절)

### 🟡 P3 - 보통 우선순위
**[이슈 제목] 개발용 보안 설정의 운영 환경 전이 가능성**
- **심각도**: Medium
- **카테고리**: 보안
- **설명**: `technical-requirements.md` 7.2절에서 `helmet` 보안 미들웨어 설정 시 `contentSecurityPolicy: false`를 개발용으로 주석 처리와 함께 명시했습니다. 이는 편리하지만, 별도의 환경 분리 구성 없이는 운영 환경에 실수로 배포될 위험이 있습니다.
- **영향**: CSP(Content Security Policy)가 비활성화된 상태로 운영에 배포될 경우, XSS 등 특정 웹 취약점에 노출될 수 있습니다.
- **개선안**: `dotenv` 또는 `cross-env`와 같은 라이브러리를 활용하여 `NODE_ENV` 환경 변수(development, production)에 따라 보안 설정이 자동으로 다르게 적용되도록 구성하는 것을 권장합니다. 예를 들어, `if (process.env.NODE_ENV === 'production') { app.use(helmet()); }` 와 같이 구성할 수 있습니다.
- **참조**: `technical-requirements.md` (7.2절)

### 🟢 P4 - 개선 항목
**[이슈 제목] Frontend-Backend 통신 데이터 형식 불일치 가능성**
- **심각도**: Low
- **카테고리**: 아키텍처
- **설명**: 본 상세설계서 8.2절에서는 Nexacro 통신 인터페이스의 입출력으로 JSON을 명시하고 있습니다. 하지만, `3. api-design.md` 기본설계서에서는 Nexacro와의 통신을 위해 상세한 XML 형식(Dataset)을 표준으로 정의하고 응답 생성 헬퍼 함수까지 제공하고 있습니다.
- **영향**: 데이터 형식에 대한 정의가 일관되지 않아, Backend API 구현 시 혼선을 유발하고 불필요한 재작업을 발생시킬 수 있습니다.
- **개선안**: 프로젝트의 표준 통신 데이터 형식을 `api-design.md`에 따라 "Nexacro Dataset XML"로 명확히 하고, 관련 문서의 내용을 일치시켜야 합니다.
- **참조**: `3. api-design.md` (3절, 10.1절)
