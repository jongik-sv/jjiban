## ⚠️ 발견된 이슈

### 🟡 P3 - 보통 해결
**[이슈 제목] 대량 데이터 처리 성능 저하 위험**
- **심각도**: Medium
- **카테고리**: 성능
- **설명**: 비기능 요구사항 `REQ-CD0300-N-001`은 10만 건의 검증 결과를 10초 이내에 표시하도록 요구하고 있습니다. 하지만 현재 상세설계에는 대량의 데이터를 그리드에 표시할 때 발생할 수 있는 성능 문제를 해결하기 위한 페이징(Paging) 또는 가상 스크롤링(Virtual Scrolling) 같은 구체적인 전략이 명시되어 있지 않습니다.
- **영향**: 수만 건 이상의 오류 데이터가 반환될 경우, 브라우저 렌더링 지연 및 UI 응답없음 현상이 발생하여 사용성을 심각하게 저해할 수 있습니다.
- **개선안**:
    1.  백엔드 API에 페이징 기능을 추가하고, 프론트엔드는 페이지 단위로 데이터를 요청하여 그리드에 표시하는 방식을 적용합니다.
    2.  Nexacro N v24에서 지원하는 가상 스크롤링 기능을 그리드에 적용하여 대량의 데이터도 성능 저하 없이 스크롤 할 수 있도록 구현합니다.
- **참조**: `docs/project/maru/10.design/12.detail-design/Task-8-2.CD0300-Frontend-UI-구현(상세설계)-gemini.md` §11

### 🟢 P4 - 개선 항목
**[이슈 제목] 비표준 단축키 사용**
- **심각도**: Low
- **카테고리**: 품질
- **설명**: UI 레이아웃 설계에서 '코드 검증 실행' 버튼의 단축키로 `F2`를 지정했습니다. `F2`는 일반적으로 '이름 바꾸기' 또는 '편집' 기능에 사용되는 표준 단축키이므로, 사용자에게 혼란을 줄 수 있습니다.
- **영향**: 사용자가 의도치 않은 동작을 경험할 수 있으며, 일반적인 UX 표준과 달라 학습 비용이 발생합니다.
- **개선안**: `F5`(새로고침/재실행) 또는 `F9`(실행)와 같이 기능에 더 부합하는 단축키로 변경하거나, 단축키 없이 버튼 클릭만으로 기능을 사용하도록 단순화하는 것을 권장합니다.
- **참조**: `docs/project/maru/10.design/12.detail-design/Task-8-2.CD0300-Frontend-UI-구현(상세설계)-gemini.md` §6.1

**[이슈 제목] 데이터셋의 RAW_DATA 필드 관련 잠재적 정보 노출**
- **심각도**: Info
- **카테고리**: 보안
- **설명**: 데이터셋 `ds_validation_result`에 `RAW_DATA` 필드가 정의되어 있습니다. 이 필드는 오류 행의 전체 데이터를 담는 용도로 보이며, 디버깅에는 유용할 수 있으나 UI에 불필요하거나 민감한 정보가 포함될 수 있습니다.
- **영향**: 의도치 않게 민감한 데이터(예: 개인정보, 내부 시스템 정보)가 사용자 화면에 노출될 위험이 있습니다.
- **개선안**: `RAW_DATA` 필드를 제거하고, 팝업에 필요한 데이터만 명시적으로 백엔드에서 받아오거나, 해당 필드를 사용하더라도 UI에 표시되는 정보는 화이트리스트 방식으로 선별하여 보여주도록 구현해야 합니다.
- **참조**: `docs/project/maru/10.design/12.detail-design/Task-8-2.CD0300-Frontend-UI-구현(상세설계)-gemini.md` §7.1
