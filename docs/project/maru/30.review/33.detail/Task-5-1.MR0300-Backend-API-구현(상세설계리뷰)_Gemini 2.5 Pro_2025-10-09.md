## ⚠️ 발견된 이슈

### 🟠 P2 - 빠른 해결 권장
**[이슈 제목] DoS 공격에 취약할 수 있는 버전 비교 API**
- **심각도**: High
- **카테고리**: 보안
- **설명**: 버전 비교 API(`GET /compare`)는 `fromVersion`과 `toVersion`을 파라미터로 받습니다. 만약 두 버전의 차이가 매우 클 경우, 모든 중간 버전을 조회하여 비교하는 로직이 구현된다면 서버에 과도한 부하를 유발하여 서비스 거부(DoS) 공격에 취약해질 수 있습니다.
- **영향**: 악의적인 사용자가 매우 큰 버전 범위를 요청하여 API 서버의 리소스를 고갈시킬 수 있습니다.
- **개선안**:
  1. 비교 가능한 최대 버전 범위를 50개 또는 100개 등으로 제한하는 정책을 추가합니다.
  2. 요청된 버전 범위가 제한을 초과할 경우, `VERSION_RANGE_EXCEEDED (-400)`와 같은 명확한 오류를 반환하도록 설계에 명시합니다.

### 🟡 P3 - 보통 우선순위
**[이슈 제목] 캐시 무효화(Cache Invalidation) 전략 부재**
- **심각도**: Medium
- **카테고리**: 아키텍처
- **설명**: 이력 조회 결과에 대해 `node-cache`를 사용한 캐싱 전략(TTL: 5분)을 언급했으나, 원본 데이터(마루 헤더, 코드, 룰 등)가 변경되었을 때 관련 이력 캐시를 어떻게 무효화할지에 대한 전략이 정의되어 있지 않습니다.
- **영향**: 데이터 변경 후 최대 5분 동안 사용자에게 이전 데이터(stale data)가 조회될 수 있어 데이터 일관성을 해칠 수 있습니다.
- **개선안**:
  - 마루 데이터에 변경이 발생할 경우, 관련된 `history:${maruId}:*` 패턴의 캐시를 모두 삭제하는 로직을 추가합니다.
  - 예를 들어, `PUT /api/v1/maru-headers/{maruId}` API 처리 시, 해당 `maruId`의 이력 캐시를 무효화하는 단계를 설계에 명시해야 합니다.

**[이슈 제목] API 에러 코드와 Nexacro 기본 에러 코드의 매핑 불일치**
- **심각도**: Medium
- **카테고리**: 품질
- **설명**: 상세설계서에는 `INVALID_DATE_FORMAT`과 같은 문자열 기반의 오류 상황을 정의했지만, `api-design.md`에서는 `-400` (파라미터 에러), `-100` (비즈니스 로직 에러) 등 숫자 형식의 `ErrorCode`를 사용하기로 정의했습니다. 두 체계 간의 매핑 규칙이 명시되지 않아 구현 시 혼란이 발생할 수 있습니다.
- **영향**: Frontend(Nexacro)에서 일관된 에러 처리가 어려워지며, 오류 코드 파싱 로직이 복잡해질 수 있습니다.
- **개선안**: 상세설계서의 모든 오류 상황에 대해 `api-design.md`에 정의된 숫자 `ErrorCode`를 매핑하는 표를 추가해야 합니다.
  - 예: `INVALID_DATE_FORMAT` → `ErrorCode: -400`, `ErrorMsg: "날짜 형식이 올바르지 않습니다..."`

### 🟢 P4 - 개선 항목
**[이슈 제목] 통계 조회 API의 잠재적 성능 저하**
- **심각도**: Low
- **카테고리**: 성능
- **설명**: 이력 통계 API(`GET /history/stats`)는 '평균 변경 주기'와 같은 데이터를 실시간으로 계산하도록 설계되었습니다. 이력 데이터가 수천 건 이상으로 누적될 경우, 이 계산은 데이터베이스에 부하를 주는 무거운 쿼리가 될 수 있습니다.
- **영향**: 이력 통계 API의 응답 시간이 점차 느려질 수 있습니다.
- **개선안**:
  - **(단기)** 통계 결과 역시 캐싱(TTL: 10분)하여 반복적인 계산을 피하도록 설계에 반영합니다.
  - **(장기)** 야간 배치(Batch) 작업 등을 통해 주요 통계 데이터를 미리 계산하여 별도의 요약 테이블에 저장하는 방안을 고려할 수 있습니다.

- **영향**: 요구사항과 상세설계 간의 중간 설계 단계가 누락되어 전체적인 설계 흐름을 파악하는 데 어려움이 있을 수 있습니다.
- **개선안**: 프로젝트 관리 절차에 따라 기본설계서 작성이 필요한지 확인하고, 필요하다면 추후 보완이 필요합니다.
